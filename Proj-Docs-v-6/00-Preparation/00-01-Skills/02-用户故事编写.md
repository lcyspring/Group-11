## 核心内容

### 2.1 什么是用户故事

用户故事是从用户视角描述功能需求的简短陈述。它不是技术规格，而是对话的占位符。

**标准格式**：

```
作为 <角色>
我想要 <功能>
以便 <价值>
```

### 2.2 用户故事示例

```
作为 平台管理员
我想要 创建新租户并配置其初始管理员
以便 新客户能够独立使用系统开展业务
```

```
作为 商城会员
我想要 通过手机浏览商品并下单
以便 方便快捷地完成购物
```

```
作为 租户管理员
我想要 查看本租户下所有员工的考勤统计报表
以便 掌握团队出勤情况并作为薪资依据
```

```
作为 海外用户
我想要 以我的母语浏览商城界面
以便 能够无障碍地了解商品信息
```

### 2.3 INVEST 原则

好的用户故事应满足：

| 原则 | 含义 | 反例 |
|------|------|------|
| **I**ndependent | 独立于其他故事 | "依赖 Story #12 完成" |
| **N**egotiable | 可协商，非固化合同 | "按钮颜色必须是 #FF0000" |
| **V**aluable | 对用户有价值 | "重构数据库连接池" |
| **E**stimable | 可估算工作量 | "优化系统性能" |
| **S**mall | 足够小，一个迭代完成 | "开发整个ERP模块" |
| **T**estable | 可测试验证 | "用户体验要好" |

### 2.4 故事拆分

史诗故事 → 用户故事 → 任务

**示例："CRM客户管理"史诗的拆分**：

```
史诗：CRM客户管理
├── Story 2.1: 客户列表查询（支持分页、搜索、多条件筛选）
├── Story 2.2: 新增客户（基本信息录入、联系人信息）
├── Story 2.3: 编辑客户信息（字段修改、状态变更）
├── Story 2.4: 删除客户（逻辑删除、确认提示）
├── Story 2.5: 客户详情（完整信息展示、关联商机列表）
├── Story 2.6: 批量导入客户（Excel 上传、解析校验）
├── Story 2.7: 客户跟进记录（添加、查看、列表）
└── Story 2.8: 导出客户列表（Excel 导出）
```

## 常见问题

**Q：用户故事和用例有什么区别？**

用户故事是轻量级的、面向对话的需求描述；用例是更正式、更详细的交互描述。在实际项目中，用户故事用于日常迭代规划，用例用于复杂场景的详细分析。

**Q：一个用户故事应该多大？**

一个用户故事应该在 1-3 天内可以完成。如果超过 3 天，说明需要拆分。

## 小结

- 用户故事格式：`作为...我想要...以便...`
- INVEST 原则保证故事质量
- 史诗需要拆分为可交付的故事
- 密讯ETM涵盖 10+ 业务模块，每个模块有独立的故事集

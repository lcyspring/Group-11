### 10.2 验收流程

```
1. 自测阶段
   ├── 开发人员自测（对照验收标准）
   ├── 代码审查 (Code Review)
   ├── 单元测试通过（mvn test）
   └── 前端 TS 检查通过（npm run ts:check）

2. 提测阶段
   ├── 提交测试申请
   ├── 部署到测试环境（Docker Compose）
   └── 提供测试账号和文档

3. 测试阶段
   ├── 功能测试（按验收标准逐项测试）
   ├── 集成测试（跨模块端到端流程）
   ├── 性能测试（JMeter 50并发）
   └── Bug 修复与回归

4. 验收阶段
   ├── 演示系统功能
   ├── 检查验收标准达成率
   ├── 审查交付物清单
   └── 签署验收报告
```

### 10.3 答辩准备

#### PPT 结构建议

```
1. 封面（1页）
2. 项目背景与目标（2页）
   - 密讯ETM的定位与价值
   - 要解决的核心业务问题
3. 技术架构（2页）
   - 15个限界上下文全景图
   - 多租户SaaS架构 + 国际化方案
4. 核心功能演示（3-4页）
   - 选取2-3个亮点模块现场Show
   - 建议: BPM审批流程 + CRM销售漏斗 + AI智能对话
5. DDD设计与技术亮点（1-2页）
   - 候选人策略引擎
   - 多租户自动隔离机制
6. 测试结果（1页）
   - 单元测试覆盖率、性能测试报告
7. 项目总结与展望（1页）
```

#### 答辩要点

**1. 讲清楚"为什么"**

不只是展示代码，要说明：
- 为什么选择 Spring Boot 3.5.x + MyBatis-Plus？
- 为什么划分为 15 个限界上下文？
- 为什么采用多租户共享数据库方案？
- 为什么自定义 Flowable 候选人策略？

**2. 准备演示环境**

- 提前部署好系统，确保可演示（Docker Compose全栈部署）
- 准备演示数据，覆盖主要业务场景
- 准备备用方案（本地环境或录屏）

**3. 密讯ETM答辩常见问题**

| 问题 | 答题思路 |
|------|----------|
| 系统最大的亮点是什么？ | 多租户SaaS架构 + 自定义BPM候选人策略引擎 + 15模块一体化平台 |
| 为什么选择多租户共享数据库？ | 运维成本低、数据集中便于报表分析、通过 TenantLineInnerInterceptor 自动隔离 |
| 国际化实现方案是什么？ | 三层: 前端 locale + 后端 messages_*.properties + 数据库 *_i18n 表 |
| 遇到了什么困难？怎么解决的？ | 具体技术问题 + 解决过程（如租户过滤与 @TenantIgnore 的场景平衡） |
| 如果重新做，会怎么改进？ | 反思 + 改进方案（如引入微服务拆分、强化 CQRS 模式） |
| 代码质量如何保证？ | 统一测试基类 + 代码规范 + Code Review + `mvn test` 卡点 |
| 系统安全性如何？ | 认证授权(Security) + BCrypt加密 + 租户隔离 + 输入校验 + XSS/SQL注入防护 |

## 常见问题

**Q：答辩时系统出 Bug 怎么办？**

不要慌，坦诚说明这是已知问题，并说明修复计划。评审更看重你对问题的理解和处理能力。

**Q：功能没做完怎么办？**

按优先级汇报已完成部分，说明未完成部分的原因和后续计划。不要试图掩盖。

**Q：PPT 应该多详细？**

PPT 是演讲的辅助，不是文档。每页 3-5 个要点，不要堆砌文字。重点放在架构图和演示效果上。

## 核心内容

### 4.1 数据库设计原则

**三大范式**：

| 范式 | 要求 | 示例 |
|------|------|------|
| 1NF | 字段不可再分 | 地址不存储为"北京市朝阳区"一个字段 |
| 2NF | 非主键字段完全依赖主键 | 订单表不包含商品名称 |
| 3NF | 非主键字段不传递依赖 | 员工表不包含部门负责人 |

**现实建议**：适度反范式化，以查询性能换取少量冗余。

### 4.2 密讯ETM数据库设计要点

#### MySQL 强制规范

- **所有SQL使用MySQL语法**: `AUTO_INCREMENT`、`ENGINE=InnoDB`、`DEFAULT CHARSET=utf8mb4`
- **禁止使用PostgreSQL**: `SEQUENCE`、`SERIAL` 等语法在项目中禁止使用
- **多租户表中必须包含 `tenant_id` 字段**

#### ER 图核心关系

### 4.3 表结构设计示例

```sql
-- 用户表设计（遵循MySQL规范）
CREATE TABLE system_user (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    username    VARCHAR(50)  NOT NULL COMMENT '用户名',
    password    VARCHAR(200) NOT NULL COMMENT 'BCrypt加密密码',
    nickname    VARCHAR(50)  COMMENT '昵称',
    email       VARCHAR(100) COMMENT '邮箱',
    mobile      VARCHAR(20)  COMMENT '手机号',
    sex         TINYINT  DEFAULT 0 COMMENT '性别: 0未知 1男 2女',
    avatar      VARCHAR(512) COMMENT '头像URL',
    status      TINYINT  DEFAULT 0 COMMENT '状态: 0正常 1禁用',
    tenant_id   BIGINT   DEFAULT 0 COMMENT '租户ID (0=平台)',
    login_ip    VARCHAR(50)  COMMENT '最后登录IP',
    login_date  DATETIME     COMMENT '最后登录时间',
    creator     VARCHAR(50)  COMMENT '创建者',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updater     VARCHAR(50)  COMMENT '更新者',
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted     TINYINT  DEFAULT 0 COMMENT '逻辑删除: 0未删除 1已删除',
    UNIQUE INDEX idx_username_tenant (username, tenant_id),
    INDEX idx_tenant_id (tenant_id),
    INDEX idx_status (status),
    INDEX idx_mobile (mobile)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统用户表';
```

### 4.4 索引设计策略

| 索引类型 | 适用场景 | 密讯ETM实际应用 |
|----------|----------|----------------|
| 主键索引 | 唯一标识 | `id BIGINT AUTO_INCREMENT` |
| 唯一索引 | 业务唯一字段 | `(username, tenant_id)` 联合唯一 |
| 普通索引 | 高频查询条件 | `tenant_id`, `status`, `create_time` |
| 联合索引 | 多条件查询 | `(tenant_id, status)`, `(tenant_id, create_time)` |
| 前缀索引 | 长字符串字段 | 邮箱前缀、URL前缀 |

### 4.5 RESTful API 设计规范

**URL 设计（管理端/admin-api + 用户端/app-api）**：

```
管理端 API：
GET    /admin-api/system/user/page            # 分页查询
GET    /admin-api/system/user/get?id={id}      # 详情查询
POST   /admin-api/system/user/create           # 新增
PUT    /admin-api/system/user/update           # 更新
DELETE /admin-api/system/user/delete?id={id}   # 删除

用户端 API：
GET    /app-api/system/user/profile            # 个人资料
PUT    /app-api/system/user/update-profile     # 更新个人资料
```

**统一响应格式**：

```json
{
  "code": 0,
  "msg": "操作成功",
  "data": {
    "id": 1,
    "username": "admin",
    "nickname": "平台管理员"
  }
}
```

**分页响应**：

```json
{
  "code": 0,
  "msg": "操作成功",
  "data": {
    "records": [...],
    "total": 100,
    "pageNo": 1,
    "pageSize": 10
  }
}
```

## 常见问题

**Q：什么时候用逻辑删除，什么时候用物理删除？**

核心业务数据（用户、订单）建议逻辑删除，方便追溯；日志、临时数据可以物理删除。密讯ETM系统中所有核心表都使用 `deleted` 字段做逻辑删除。

**Q：API 为什么分 /admin-api 和 /app-api？**

管理端 API 面向后台管理用户，需要完整权限校验；用户端 API 面向 C 端用户（如商城），采用更轻量的鉴权。分离部署时可通过不同网关策略进行流量管控。

**Q：数据库必须用MySQL吗？**

是的。密讯ETM项目开发、测试、生产环境统一使用 MySQL，不得使用 PostgreSQL。所有 SQL 脚本均使用 MySQL 语法。

## 小结

- 数据库设计遵循三范式，适度反范式化
- 所有SQL使用MySQL语法，禁止PostgreSQL
- RESTful API 统一 `/admin-api` 和 `/app-api` 双域
- 统一响应格式 `{code, msg, data}` 是项目铁律

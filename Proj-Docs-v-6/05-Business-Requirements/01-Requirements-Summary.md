## 1. 项目目标

基于 MITEDTSM 平台和 MIT-FMP 基础架构，构建完整的 **CRM + OA** 融合管理系统，覆盖客户关系管理、销售管理、财务管理、工单服务、营销工具和办公自动化。人力资源管理作为P2增值模块，在CRM核心功能交付后根据余力投入。

### 1.1 核心目标

- 构建客户全生命周期管理平台，覆盖线索、商机、订单、回款全流程（对照 SRS M11-M12）
- 构建灵活可配置的审批流引擎，支撑CRM各类审批场景（对照 SRS M13）
- 提供多维度的数据分析与决策支持能力，以CRM报表为优先（对照 SRS M14）
- 构建可配置的权限体系和数据字典（对照 SRS M03-M04）
- 构建CRM办公协作平台，支撑日常办公协同（对照 SRS M15）
- HRM功能（M06-M10）作为P2增值模块，CRM核心交付后有余力时投入

### 1.2 业务价值

| 价值维度 | 描述 |
|----------|------|
| 效率提升 | 自动化审批流、批量数据处理、定时任务减少人工操作 |
| 数据统一 | HR与CRM数据互通，消除数据孤岛 |
| 决策支持 | 销售漏斗、人事报表、财务统计等多维度分析 |
| 合规保障 | 完整操作审计日志、权限精细控制、数据脱敏 |
| 客户体验 | 客户360度视图、客户关怀自动化、工单SLA管理 |

## 2. 核心业务流程

### 2.1 销售全链路 (主流程)

```
线索导入 → 创建客户 → 创建商机 → 产品报价 → 阶段推进
    → 成交 → 创建订单 → 提交审批 → 审批通过 → 回款 → 开票
```

**涉及模块**: 线索管理(M12) → 客户管理(M11) → 商机管理(M12) → 订单管理(M12) → 审批流程(M13) → 财务管理(M12)

### 2.2 客户生命周期

```
新客户(潜在) → 跟进维护(意向) → 成交客户 → 持续服务
                      ↓
              超30天无跟进 → 掉入公海 → 被他人领取
                      ↓
              无效客户 → 标记垃圾客户
```

**涉及模块**: 客户管理(M11)

### 2.3 审批链路

```
任何需审批业务 → 提交申请 → BPM流程启动(Flowable)
    → 审批人审批(通过/驳回/转办/加签) → 通过后业务生效
    → 超48小时未审批 → 自动催办提醒
```

**涉及模块**: 审批流程(M13)，覆盖请假/出差/借款/请示/报销/拜访/回款/订单/退款共9类审批

### 2.4 员工全生命周期

```
招聘录用 → 办理入职(生成工号+账号) → 试用期 → 转正
    → 日常(考勤/绩效/薪酬) → 调岗/晋升 → 离职(账号禁用)
    → (30天后)返聘 → 恢复账号
```

**涉及模块**: 招聘管理(M06) → 员工管理(M07) → 考勤管理(M08) → 绩效管理(M09) → 薪酬管理(M10)

### 2.5 工单服务流程

```
客户提出问题 → 创建工单 → 自动分配处理人 → 处理中
    → 已完结(记录方案+耗时)
    → 信息不足 → 退回发起人补充 → 重新提交
    → 超SLA时限 → 自动升级提醒
```

**涉及模块**: 工单域(WO) / 办公协作(M15)

### 3.2 优先级说明

| 优先级 | 含义 | 交付策略 |
|--------|------|----------|
| **P0 (Must Have)** | MVP 必须功能，无此功能系统不可用 | Sprint 1-5 交付 |
| **P1 (Should Have)** | 重要但有替代方案，影响体验但不阻断核心流程 | Sprint 6-8 交付 |
| **P2 (Could Have)** | 增值功能，可后续迭代 | Sprint 9+ 交付 |

## 5. 非功能性需求

### 5.1 性能要求

| 类别 | 指标 | 度量标准 |
|------|------|----------|
| API响应 | 列表查询 | P95 < 500ms |
| API响应 | 报表/分析 | P95 < 3s |
| 页面加载 | 首次加载 | ≤ 3秒 |
| 数据处理 | Excel导入(≤1000行) | < 30s |
| 批量处理 | 群发(1000条) | < 5min |
| 并发支持 | 同时在线用户 | ≥ 120人 |

### 5.2 可用性要求

| 类别 | 要求 |
|------|------|
| 系统运行时间 | 99.5% (月度) |
| 故障恢复 | 自动故障转移，RTO < 30min |
| 数据备份 | 每日全量备份，保留30天 |

### 5.3 安全要求

| 类别 | 要求 |
|------|------|
| 认证机制 | Token认证 (Sa-Token)，有效期2小时，支持刷新 |
| 授权控制 | RBAC + 方法级权限 (@PreAuthorize)，精确到按钮级别 |
| 数据隔离 | 多租户 tenant_id 拦截器 |
| 传输加密 | TLS 1.3 |
| 密码存储 | BCrypt加密 |
| 敏感数据 | 薪酬数据脱敏展示，独立密码验证 |
| Web安全 | OWASP Top 10 防护 (XSS/CSRF/SQL注入)，复用框架层 |
| 审计日志 | @BizLog 全量操作日志 |

### 5.4 可扩展性要求

| 类别 | 要求 |
|------|------|
| 架构 | 无状态服务，支持水平扩展 |
| 国际化 | 中文/英文 (Phase 1)，可扩展至阿拉伯语等8语言 |
| 浏览器兼容 | Chrome/Edge/Firefox 最新两个版本 |
| 部署 | 单体架构，支持后续微服务拆分 |

### 5.5 国际化要求

| 层面 | 实现方式 |
|------|----------|
| 前端Web | `src/locales/<lang>/` 语言包 |
| 前端Mobile | `src/locale/` 语言包 |
| 后端 | `messages_<lang>.properties` 资源文件 |
| 数据库 | `system_menu_i18n` 菜单翻译表 |

## 10. 验收标准总则

### 10.1 功能验收标准

- 所有 User Story 描述的验收标准必须全部通过
- 所有 API 接口通过 Knife4j 文档可测试
- 所有页面功能可正常操作，无阻断性缺陷
- 核心业务流程可端到端走通

### 10.2 性能验收标准

- 页面首次加载时间 ≤ 3 秒
- API 接口响应时间（P95）≤ 500ms（列表）/ 3s（报表）
- 支持 120 用户并发操作

### 10.3 安全验收标准

- 所有 API 需通过 Token 认证
- 密码必须 BCrypt 加密存储
- 敏感操作需记录 @BizLog 操作日志
- 权限控制需精确到按钮级别
- 租户数据隔离有效

### 10.4 文档验收标准

- 提供完整的 Knife4j 接口文档
- 提供用户操作手册
- 提供部署运维手册

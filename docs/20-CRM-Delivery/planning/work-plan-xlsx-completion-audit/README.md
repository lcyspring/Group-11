# 工作计划表完成情况审计

审计日期：2026-07-17。分支：`develop`。

## 结论

工作计划表包含 `7.14`、`7.16`、`7.17`、`7.18`、`7.19` 五个工作表，共 386 条非空计划项。
表内只有 1 条描述完全相同，但不同组大量使用“基础 CRUD”“设计表结构”“开发接口”“前端联调”
等不同措辞重复描述同一能力，因此不能按原始行数计算完成率。

本次按业务对象、命令、状态机、页面和验收目标合并为 65 个能力簇，并同时检查代码、数据库迁移、
自动化测试、运行记录和现有交付文档。结果如下：

| 状态 | 能力簇数 | 占比 | 含义 |
|---|---:|---:|---|
| 完成/复用完成 | 58 | 89.23% | 目标能力存在，且有实现与测试或运行证据 |
| 部分完成 | 4 | 6.15% | 主体能力已存在，但计划表中的增强项没有全部完成 |
| 未形成交付证据 | 3 | 4.62% | 没有找到该专项产物或正式验收证据 |

若将“部分完成”按一半权重估算，总体约为 **92.31%**。这比原始 386 行逐条统计更接近真实状态。
此前 62/62 的结论仍适用于已确认的 CRM/OA 核心业务功能表；本次新增差距来自工作计划表中的
高级导入导出、点击分析、专项压测、UAT 签署和跨域规范等更细交付项，两者口径不同。

因此当前不是“还有大型业务模块没有开发”，而是核心闭环完成后的增强、验收和文档收口。现有显式
YAML、幂等数据库迁移、Ubuntu 工具链镜像和 rootless Podman 部署已经提供稳定底座；后续 Provider
provision、压测参数和开关都可以继续显式配置，不需要推翻现有构建部署流程。

## 真实剩余项

1. 客户导出已有对象权限与受保护下载，但没有异步导出任务、任务状态和一次性下载授权模型。
2. 营销已有短信送达/失败、邮件接受/失败/首次打开与比率；计划表要求的链接点击事实和点击率未实现。
3. YAML 当前只显式选择 `record-only/system`、追踪和同步策略；真实短信渠道与邮件账号仍来自系统数据库，
   尚未形成由 `deploy.sh` YAML 完整声明并安全 provision 的账号配置，生产供应商验收也依赖外部账号。
4. 工单已有权限负向、状态机、SLA、统计和性能基线，但没有找到独立的 SQL 注入/XSS 专项结果、
   “50 并发 JMeter”原始方案与报告。
5. 没有找到工单 16 条业务 UAT 的逐项签署，以及前端/后端/数据库/测试四方签署和知识转移记录。
6. 没有找到计划表要求的支付流水、跨域回调、权限互通三部分 CRM/OA/营销集成规范文档。

完整矩阵见 [01-去重能力矩阵.md](./01-去重能力矩阵.md)，剩余工作的验收边界见
[02-剩余工作与优先级.md](./02-剩余工作与优先级.md)，证据索引见 [03-证据索引.md](./03-证据索引.md)。

## 判定边界

- `7.18`、`7.19` 虽然在审计日之后，仍按用户要求全部纳入对比；尚未到计划日期不自动视为完成。
- 同一对象的建表、DO、Mapper、Service、Controller、API、页面、联调合并为一个完整能力簇。
- “复用现有系统能力”在计划本身允许复用且运行入口存在时记为复用完成，不重复建设。
- 外部 Provider 的适配接口和显式配置可由代码完成；真实供应商账号与生产验收不能由本仓库伪造。
- “完成”不等于永远无缺陷；后续浏览器验收发现的问题继续进入独立 Bug 日志。

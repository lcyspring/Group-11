# 剩余工作与优先级

以下均是核心 62 项业务闭环之外的增强或验收收口，不代表 CRM 主要模块未完成。实施时继续沿用显式
YAML 和幂等 provision：命令行只指定配置文件路径，环境差异进入 YAML，本地秘密不写入仓库。

## 已关闭增强

### 客户导入预检与字段映射（2026-07-17）

已形成“上传文件 → 解析表头 → 映射系统字段 → 预检错误/重复 → 用户确认 → 后台导入”的显式任务。
预检不得写业务表；确认命令应携带预检任务版本，避免文件被替换后使用旧确认结果。

验收要求：必填、类型、长度、字典、负责人权限、名称/手机号重复均返回行号；同一预检任务重复确认幂等；
失败行可下载，成功/失败数量可追溯。

## P1：补齐产品能力

### 1. 异步导出任务与下载授权

建议增加导出任务真源，保存对象类型、筛选快照、授权范围快照、状态、结果文件、过期时间和失败原因。
生成与下载都必须再次验证租户和对象权限；大数据量不能占用 HTTP 请求线程直至完成。

验收要求：排队/运行/成功/失败/过期五态；同一用户只能读取自己的任务，管理员也不能跨租户；
下载令牌短期有效且不可猜测，结果文件过期可清理。

### 2. 营销链接点击分析

在现有邮件打开令牌之外增加每收件人、每链接的不可猜测跳转令牌。匿名端点只记录首次/累计点击并
302 到经过协议和域名白名单校验的目标地址，不得形成开放重定向。

验收要求：点击数、独立点击人数、点击率分母口径明确；重复点击不改变首次点击时间；无效令牌不泄漏；
退订链接和普通营销链接分开统计。

### 3. Provider 账号显式 provision

现有 `crm_marketing.provider_mode` 只决定 `record-only` 或调用 System 短信/邮件服务，真实渠道和账号仍
依赖数据库中的短信渠道、邮件账号与模板。应在 `deploy.sh` 使用的本地 YAML 中显式声明是否 provision、
渠道类型、端点、账号引用和模板映射；示例 YAML 只保留非秘密占位符，真实秘密由本地未跟踪配置或
secret 文件提供。重复部署必须幂等，关闭 provision 时不得覆盖生产数据库已有配置。

状态：已关闭。运行 YAML 已提供 `disabled/create-only/managed` 三种所有权模式，短信渠道/模板与邮件
账号/模板四个聚合已通过隔离库 4/4 回归；示例仍为占位值。真实发送验收继续列为外部账号依赖，
不把数据库 provision 误标为供应商送达。

## P2：补齐专项验收证据

### 4. 工单专项性能与安全

- 增加可复现的 50 并发场景，覆盖创建、领取/分派、状态流转、SLA 扫描与统计读取；
- 保存脚本、数据规模、硬件/容器资源、预热、吞吐、错误率、p95/p99 和清理结果；
- 增加对象越权、SQL 注入载荷、存储/反射 XSS、非法状态和批量参数边界用例。

已有通用 CRM 性能和运行安全基线可以复用，但不能替代计划表点名的工单场景原始证据。

### 5. 工单 UAT 与交接

把 16 条业务 UAT 固化为可重复用例，签署至少包含前端、后端、数据库和测试四类责任结论；知识转移
应附操作指南、故障定位、数据清理、回滚和已知限制。签署是人员协作事项，不能由代码提交伪造。

### 6. 支付跨域集成规范

补充独立中文文档，至少明确：

- 在线支付流水与 CRM 回款/退款事实的边界和映射键；
- 支付回调的签名、幂等、重放保护、失败补偿和跨租户隔离；
- OA 报销/借款、CRM 回款/退款、营销付费活动是否触发支付，以及禁止的反向依赖；
- API 入参/出参、错误码、状态映射、权限矩阵和审计字段；
- 当前“不集成”的场景也要显式写明，避免误把 `pay_refund` 当成 CRM 退款。

## 外部依赖

真实供应商仍需要账号、沙箱/生产地址、签名与模板审批、送达回执协议。仓库可以先完成安全的 YAML
provision 契约，但拿到外部信息后才能做真实 Provider 验收；不得把 `record-only` 或本地记录标记成
真实送达。

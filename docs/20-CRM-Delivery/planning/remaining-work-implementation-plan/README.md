制定日期：2026-07-17。实施分支：`develop`。

## 目标与边界

核心 CRM/OA 62 项业务闭环保持完成状态。本 Plan 只处理工作计划表复核后留下的增强、专项验收和
文档收口，不重新复制客户、订单、财务、工单等既有聚合。

统一工程约束：

- 命令行只接收 YAML 配置路径，环境差异、开关和资源上限进入显式 YAML；
- 编译、测试和前端产物统一使用 Ubuntu 26.04 Podman 工具链；
- 数据库变更必须幂等并进入 `database/manifests/mysql-compatibility.manifest`；
- 每项建立独立功能、测试、覆盖率和 Bug 日志目录；
- 每阶段完成后单独提交，提交说明使用 `Where : feat/change description`；
- 不推送远端，服务持续保持可观察。

## P1-A：客户导入预检设计

### 数据模型

引入导入预检聚合，保存租户、创建人、原文件名、字段映射、负责人、是否允许更新、预检状态、
总行数、可创建数、可更新数、失败数、行数据快照、过期时间和确认时间。预检行数据保存规范化后的
业务字段及逐行结论，不保存原始文件二进制。

状态固定为：`PREVIEWED`、`CONFIRMED`、`EXPIRED`。确认命令只允许从 `PREVIEWED` 原子进入
`CONFIRMED`；相同任务重复确认返回原结果，不重复创建或更新客户。

### 接口

- `POST /crm/customer/import-preview`：上传 Excel 和字段映射，仅解析、校验和查重；
- `GET /crm/customer/import-preview/get`：读取本人当前租户预检结果；
- `POST /crm/customer/import-preview/confirm`：按预检快照确认导入；
- 保留既有 `/crm/customer/import` 兼容入口，在新页面切换完成后标注为兼容 API，不立即删除。

### 字段映射

系统字段固定白名单：客户名称、手机、电话、QQ、微信、邮箱、地区、详细地址、所属行业、客户等级、
客户来源、备注。首次上传自动识别标准表头；非标准表头由用户映射。客户名称必须且只能映射一次，
同一个文件列不能映射到多个系统字段。字典值既支持标签，也支持合法数值；地区支持标准路径或 ID。

### 验收场景

1. 标准模板自动映射并显示创建/更新/失败数量；
2. 自定义表头在用户映射后预检成功；
3. 空名称、超长名称、非法邮箱/字典/地区、文件内重复和数据库重复返回行号与原因；
4. 预检不产生客户、权限或负责人历史；
5. 确认后创建/更新、权限和负责人历史与既有导入规则一致；
6. 跨用户、跨租户、过期任务和被篡改映射无法确认；
7. 同一任务重复确认不产生第二次写入。

## 后续阶段设计摘要

P1-B 使用独立导出任务表和受保护文件引用，任务提交时冻结筛选与授权范围，生成和下载时再次校验；
P1-C 在现有营销收件人事实旁增加链接及点击事实；P1-D 复用当前非执行式 YAML 解析器和数据库
初始化容器，不把 shell `eval` 或秘密写入仓库；P2 阶段只补可复现证据和跨域契约，不为签署结果
造假。

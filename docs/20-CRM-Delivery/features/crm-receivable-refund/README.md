# CRM-FEATURE-033：回款退款与业务冲销闭环

## 完成内容

- 新增独立于支付渠道 `pay_refund` 的 CRM 回款反向业务模型；
- 支持“客户退款”和“业务冲销”两种业务类型，以及草稿、审批中、通过、驳回、取消五态；
- 新增退款主表和不可变动作轨迹表，保存原回款、客户、合同、负责人和流程实例快照；
- 提供草稿创建、修改、删除、分页、详情、提交审批、修订和可退金额摘要接口；
- 审批回调按当前流程实例和目标状态幂等，旧流程或重复事件不重复记账；
- 管理端提供列表、表单、详情、审批详情、修订、提交和动作轨迹入口，并同步中、英、阿三语文本；
- 流程定义键和业务编号前缀由 YAML 显式配置，命令行仍只接收配置文件路径。

## 金额与并发不变量

- 只允许关联审批通过的原回款；
- 退款/冲销金额必须为正，原因必须为 10～500 字；
- 审批中与已通过记录共同占用原回款可退额度；
- 提交时先锁原回款，再重新汇总占用金额，防止两个草稿并发超额；
- 更新接口只允许修改类型、日期、金额、原因和备注，原回款及客户、合同、负责人快照不可篡改；
- 已提交记录不可物理删除，驳回或取消后必须显式修订回草稿再提交。

## 权限设计

- 创建时继承原回款负责人并写入退款对象 OWNER 权限；
- 查询复用 CRM 对象权限，缺省场景对普通用户收敛为“我负责”，CRM 管理员保留全量；
- 表单远程检索并显式合并“我负责”和“我参与”的已审批回款，按 ID 去重且保证当前回款回显；
- 提交、修订和删除均在 Service 层校验对象写权限，不依赖前端按钮隐藏。

## 配置与数据库

- 配置：`mitedtsm.crm.receivable-refund.process-definition-key`；
- 配置：`mitedtsm.crm.receivable-refund.number-prefix`；
- 迁移：`database/migrations/new-crm-receivable-refund.sql`；
- 兼容迁移可重复执行，并由 Podman YAML 指定的 manifest 统一加载。

## 当前边界

本功能已经完成代码、数据库、前端、单元测试、构建和真实创建/删除/超额拒绝验收。当前运行库尚未部署
`crm-receivable-refund-audit` 流程定义，因此不能宣称真实审批提交链已经通过。没有审批人规则时不擅自创建
形式化流程；流程部署后仍需补一次提交、通过、驳回、取消和旧回调运行验收。

本台账不等同银行退款、会计总账或在线支付渠道退款；报销/费用真源仍是独立后续工作。

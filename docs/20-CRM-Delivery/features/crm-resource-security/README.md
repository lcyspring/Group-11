# CRM-FEATURE-034：文件、附件和导出对象权限加固

## 完成内容

- 线索、联系人、客户、商机、合同、回款、回款计划和发票八类导出统一执行对象级校验；
- CRM 管理员可导出，当前用户或其下属拥有 OWNER/WRITE 时可导出，只有 READ 或无权限时拒绝整批导出；
- 修复联系人导出误写页码、合同导出未取消分页造成的数据截断；
- 合同附件改为“专用上传—受管文件绑定—授权下载”，生命周期响应不再暴露原始文件 URL；
- 合同上传要求对象 WRITE，下载要求对象 READ，附件 ID 必须属于请求合同；
- 公共 Infra 文件路由对 YAML 配置的受保护目录返回 404，并规范化 `/`、`\\` 和 `..` 路径后再判断；
- 前端通过带认证的合同附件 API 上传、下载，不再直接打开匿名文件地址。

## 显式配置

```yaml
mitedtsm:
  infra:
    file-security:
      protected-path-prefixes: [crm-protected/]
  crm:
    security:
      protected-file-directory: crm-protected/contract
```

命令行只传 `podman/config/verify-crm-resource-security-ubuntu-26.04.kdl`。Ubuntu 构建容器始终执行
仓库当前版本的入口脚本，缓存镜像只提供 JDK、Maven、Node 和 pnpm 工具链。

## 安全不变量

- 页面查询权限不自动等于批量导出权限；只读协作者不能通过导出取得完整离线副本；
- 业务附件必须对应 `infra_file` 中的受管文件，客户端不能绑定任意外部 URL；
- 文件实际路径必须落在该合同的 `crm-protected/contract/{contractId}` 目录，文件名、类型和大小以服务端元数据为准；
- 公共文件路由不能读取受保护目录，业务下载必须先经过合同对象权限与附件归属校验；
- 任一导出对象不满足权限时整批失败，不产生部分成功且难以审计的文件。

## 当前边界

新上传合同附件已经进入受保护目录。历史附件虽然不再由生命周期接口返回原始 URL，但已被外部记录的旧
公共 URL 仍可能有效；完整关闭它们需要逐个复制到受保护目录、更新附件引用并校验摘要，不能只修改数据库
路径。本批将其保留为显式迁移风险，不伪称历史对象已经物理迁移。

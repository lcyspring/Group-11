# 客户垃圾池完整状态机

- 分支：`develop`
- 状态：已实现
- 配置：`mitedtsm.crm.pool-policy.garbage`

本功能在客户 `OWNED/PUBLIC/GARBAGE` 显式池状态上建立管理员隔离区。垃圾池不是“负责人
为空”的另一种公海，而是只能由 CRM 管理员查询和操作的终止前隔离状态。

## 状态与动作

| 当前状态 | 动作 | 结果 |
|---|---|---|
| PUBLIC | 管理员填写原因并转入垃圾池 | GARBAGE，清除客户及联系人全部团队权限 |
| PUBLIC | 达到 YAML 滞留期限和累计入池次数 | 自动转入 GARBAGE |
| GARBAGE | 管理员恢复 | PUBLIC，不直接分配负责人 |
| GARBAGE | 管理员永久删除 | 无下游引用时物理删除客户主体 |

手工和自动转入均在客户行锁内复核状态。锁定、已成交、存在活跃商机或受保护合同的客户
不得转入。自动迁移采用主键游标、批次上限、最大批次数和集群调度锁，不使用无限扫描。

## 永久删除保护

永久删除统一检查下级客户、联系人、商机、合同、已转换线索、回款计划、回款、发票、
退款/冲销、报销和客服工单。任何有效引用都返回具体引用类型，不执行部分删除。客户主体
使用租户、主键、GARBAGE 状态和空负责人联合条件物理删除；跟进记录按现有删除契约清理，
归属和生命周期轨迹作为审计记录保留。

客户存在联系人时仍先返回引用保护。为避免“唯一首联系人禁止删除”与“永久删除要求无
联系人”形成死锁，仅 CRM 管理员可以在 GARBAGE 状态删除唯一首联系人；在管和公海客户
继续保持至少一个首联系人的原有不变量。垃圾池联系人权限已被隔离，删除后的权限清理使用
幂等语义。

## 管理端

- 垃圾池页面支持名称、手机、等级和来源筛选；
- 展示前负责人、累计入池次数、垃圾时间和原因；
- 客户名称可进入详情，但详情不会错误展示领取、分配或放入公海按钮；
- 公海列表提供填写原因后转入垃圾池的操作；
- 恢复后的公海原因显示为“从垃圾池恢复”；
- 中文、英文和阿拉伯文文案同步。

## 显式配置

`expire-days`、`minimum-pool-cycles`、`batch-size`、`max-batch-size` 和 `max-batches` 全部来自
YAML，并校验批量不得超过部署安全上限。命令行仍只负责选择 YAML 文件路径。

永久删除当前定义为 CRM 业务主体不可恢复，不等同于法律意义的数据主体抹除；审计轨迹
的保留期限应由后续统一数据留存策略治理。

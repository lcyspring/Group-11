# CRM-FEATURE-019：商机赢单幂等转合同

更新日期：2026-07-14

## 决策来源

- 已批准的 `ADR-002`～`ADR-006`；
- `US-OPP-004`、`GAP-OPP-003`、`GAP-CTR-001/002`；
- 决策包：`docs/20-CRM-Delivery/decisions/crm-sales-model/`。

## 本次实现

- 新增 `POST /crm/contract/create-from-business` 专用转换入口；
- 只有赢单商机允许转换，活动、输单和无效商机均由后端拒绝；
- 转换入口同时要求 `crm:contract:create` 功能权限和来源商机 WRITE 对象权限；
- 服务端强制继承商机客户和负责人，忽略请求篡改值；
- 前端仅在赢单且尚无合同时显示“从赢单商机创建合同”；
- 表单自动预填商机名称、客户、负责人、下单日期、折扣和产品报价；
- 重复请求返回已有合同 ID，不重复生成编号、产品行或 OWNER 权限；
- 新增 `source_business_id` 和租户级有效转换唯一键，兜住并发首次转换；
- 逻辑删除后允许重新转换；
- 历史上同一商机关联多份合同的数据不回填、不删除、不合并；
- Podman MySQL 初始化显式执行新迁移。

## 幂等和兼容策略

历史 `business_id` 只表示普通关联，可能一对多。新字段 `source_business_id` 只标记正式
转换来源，生成列在 `deleted=0` 时形成 `tenantId#businessId` 唯一键。转换前若已存在历史
有效关联合同，返回最早一份，避免再创建；全新转换则由数据库保证只保留一份。

## 阶段边界

本批创建合同草稿，不自动提交 `crm-contract-audit`、不创建 ERP 履约订单。产品行已保存
交易价格和数量，但产品名称、规格、单位等完整不可变文本快照仍是 `GAP-OPP-002` 的后续
完善项。合同驳回/取消后的修订与重提已由后续 `CRM-FEATURE-020` 完成。

本批关闭安全缺陷 `CRM-CORE-BUG-020`；测试基础设施问题记录为 `TEST-BUG-008`。

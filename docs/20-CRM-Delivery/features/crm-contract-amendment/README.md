# CRM-FEATURE-049：已签署合同补充协议独立审批闭环

完成日期：2026-07-16。关联：GAP-CTR-001/002、GAP-ORD-003、GAP-APR-002、ADR-015。

## 交付结论

已签署合同不再通过普通更新覆盖历史。正式业务变更使用独立补充协议聚合，保存合同和产品的
变更前后快照，经独立 BPM 流程审批通过后，才把目标版本投影到原合同并锁定补充协议依据附件。

## 数据模型与不变量

- `crm_contract_amendment` 保存编号、客户端幂等请求号及哈希、基线/目标版本、审批状态、流程实例、
  生效时间、前后合同与产品快照、变更前后金额及差额；
- 仅审批通过且存在有效实际签署事实的合同允许创建补充协议；
- 同一合同任一时刻只允许一个未结束补充协议，目标版本必须等于基线版本加一；
- 草稿候选版本不推进当前有效版本，只有审批通过动作推进版本；
- 合同目标金额不得低于已审批回款、回款计划总额和蓝票减红票后的净开票金额；
- 已有产品行继承不可变目录快照和税率，按补充协议折扣后的净额重新计算税额与含税金额；
- 重复产品行、跨合同产品行、跨合同补充协议、跨租户读取和过期审批回调均失败关闭；
- 审批终态条件更新、合同/产品投影和附件锁定处于同一事务，任一步失败整体回滚。

## 生命周期

补充协议动作写入合同不可变轨迹：创建、修改、提交、审批通过生效、驳回和取消。驳回或取消后
可以修订并重新提交；审批通过后记录生效时间，候选合同字段、产品行和金额一次性成为当前版本。

补充协议依据附件使用合同受保护目录，并显式绑定 `contractId`、`amendmentId` 和目标版本。
进入审批后不能删除，审批通过后标记为不可变。

## 接口与前端

- `POST /crm/contract-amendment/create`：创建幂等草稿；
- `PUT /crm/contract-amendment/update`：修订草稿、驳回或取消记录；
- `PUT /crm/contract-amendment/submit`：校验依据附件和财务下限后提交独立流程；
- `GET /crm/contract-amendment/get`：返回可编辑目标投影，不泄露原始 JSON 快照；
- `GET /crm/contract-amendment/list`：按合同展示版本、金额差额和审批生命周期。

合同详情“签署与版本”区域已提供创建、编辑、产品行维护、附件上传/删除、提交审批和列表展示，
并补齐中文、英文和阿拉伯文文本。

## 显式配置

```yaml
mitedtsm:
  crm:
    contract-amendment:
      process-definition-key: crm-contract-amendment-audit
      number-prefix: BC
```

流程模型、审批角色和验收参数同样由 YAML 提供；脚本命令行只接受配置文件路径。

## 明确边界

- 不接入或伪造外部电子签平台能力；补充协议审批与电子签供应商是两个独立边界；
- 不允许补充协议绕过既有回款、计划和发票事实降低合同金额；
- 不物理覆盖历史快照，也不把未审批候选版本暴露为当前有效版本。

# 商机输单与无效闭环

## 差异来源

- 原型 Story：`US-OPP-005`；
- 差异：`GAP-OPP-001`；
- 现有资产：`crm_business.end_remark` 和 `CrmBusinessDO.endRemark` 已存在，
  但状态接口、服务和页面没有形成采集与展示闭环。

## 本次完成

- 输单、无效时结束原因必填，去除首尾空白后至少 10 字、最多 500 字；
- 赢单不要求原因，并清空请求中可能携带的陈旧原因；
- 阶段状态与结束状态必须二选一，拒绝歧义请求；
- 条件更新同时比较旧阶段与旧结束状态，避免并发覆盖；
- 结束商机时保留最后一个销售阶段；
- 状态条件更新同时持久化 `end_remark`；
- 商机详情展示结束原因；
- 前端弹窗按输单/无效动态展示和校验原因；
- Service 接口增加 `@Valid`，内部代理调用也执行请求约束；
- 补齐中、英、阿三语文案。

## 状态约束

| 请求 | 结果 |
|---|---|
| 仅 `statusId` | 推进活动阶段 |
| 仅赢单 `endStatus` | 结束商机，原因不保存，保留最后阶段 |
| 仅输单/无效 `endStatus` + 合法原因 | 结束商机并保存原因，保留最后阶段 |
| 同时提交 `statusId` 和 `endStatus` | 参数校验失败 |
| 并发状态已经变化 | 返回 `1020002004`，要求刷新重试 |

## 后续闭环

本次关闭“原因采集—校验—持久化—展示—并发保护”子闭环。原型要求的
`OpportunityLostEvent` 下游消费、赢单到合同/订单的幂等转换仍受 ADR-002/004
约束，不能在转换目标未确定时提前硬编码。

测试证据见 `docs/20-CRM-Delivery/testing/business-loss-closure/`；缺陷详情见
`CRM-CORE-BUG-015` 至 `CRM-CORE-BUG-018`。

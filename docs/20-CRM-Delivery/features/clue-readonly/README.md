# CRM-FEATURE-006：已转换线索只读保护

## 需求证据

- `docs/03-Gap-Analysis/03-Business-Domain-Gaps.md` 的 `GAP-LEAD-002` 明确列出“转换后只读”尚未冻结和实现。
- `docs/03-Gap-Analysis/02-End-to-End-Traceability-Matrix.md` 要求线索转换具备幂等、失败回滚和来源/目标审计。
- 原型转换语义是把后续经营活动切换到客户，线索保留为来源历史，不应继续形成另一套可变事实。

## 只读边界

线索转换成功后，以下写入口统一返回 `1020001002`：

- 编辑线索字段；
- 删除线索；
- 转移负责人；
- 新增或删除线索跟进记录；
- 添加、编辑、移除或自行退出线索团队权限。

读取线索详情、历史跟进、团队成员、操作日志及导出不受影响。前端列表和详情同步隐藏编辑、删除、转移、跟进及团队操作，但服务端校验才是最终约束。

## 并发模型

- Mapper 以 `SELECT ... FOR UPDATE` 当前读锁定线索。
- 转换、更新、删除、转移、跟进和团队权限写操作均在事务内先取得同一行锁。
- 转换先取得锁：后续写操作看到 `transform_status=true` 并拒绝。
- 写操作先取得锁：转换等待其提交，再以当前读取得最新字段创建客户，不使用等待前的 RR 旧快照。
- 原有条件更新仍保留为转换状态的防御性原子校验。

## 关联修复

- 跟进创建增加事务；插入跟进后业务对象校验失败时，已插入记录随事务回滚。
- 权限更新校验每个权限 ID 必须属于请求的 `bizType/bizId`，阻止伪造另一个可管理对象来修改目标权限。
- 权限 ID 列表由 `@NotEmpty` 拒绝空数组。

## 完成证据

- 线索专项 3/3、权限专项 2/2、CRM 全量 42/42。
- Rootless Podman 真实只读与权限负向验证 10/10。
- MySQL 行锁双向排队时序验证 2/2。
- ESLint、Web `build:prod`、Server 生产 JAR 和 Podman Server/Web/Mall 均通过。
- 测试线索、客户、联系人、跟进、日志及权限清理后均为 0。

详细测试与覆盖率见 `docs/20-CRM-Delivery/testing/clue-readonly/`。

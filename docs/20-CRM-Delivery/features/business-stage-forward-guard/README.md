# CRM-FEATURE-018：商机阶段前向推进与审计说明

更新日期：2026-07-14

## 需求来源

- `Proj-Docs-v-6`：`OPP-004`、`BC-03`、`TC-OPP-001`；
- `US-OPP-004`：只允许确认的状态迁移，每次迁移记录前后阶段和操作者；
- `GAP-OPP-001`：完善商机阶段、回退规则和权限；
- 原型商机详情的阶段推进与操作日志。

## 现状问题

原页面把同一状态组的全部阶段放入下拉框，后端仅校验目标阶段属于该状态组，因此可以
从高排序阶段回退到低排序阶段，违背已明确的“只能向前推进”规则。普通阶段变化也没有
要求推进说明，操作日志只能看到前后状态，无法说明推进依据。

缺陷单独记录为 `CRM-CORE-BUG-019`。

## 本次实现

- 前端根据状态 `sort` 只展示当前阶段之后的候选阶段；
- 后端重新读取并校验当前、目标阶段，拒绝目标排序小于或等于当前排序；
- 重复选择当前阶段保持原有“状态未变化”错误语义；
- 普通阶段推进要求填写非空说明，最多 500 个字符；
- 推进说明在服务层去除首尾空白；
- 操作日志追加推进说明，现有日志继续记录操作者、时间、前后阶段；
- 输单/无效原因也追加到同一状态操作日志，赢单不伪造说明；
- 终态的并发 compare-and-set、输单原因和最后阶段保留逻辑保持不变；
- 中、英、阿三语文案齐备。

## 阶段边界

两套需求对“跳过中间阶段”存在冲突：一处要求禁止跳过，另一处要求警告确认。因此本批
只落实没有冲突的“不得回退”不变量；向前跨级仍保持允许，待业务签署后再决定禁止还是
二次确认。

商机赢单后创建合同、CRM 销售订单还是 ERP 履约订单继续受 `ADR-002/004/005` 阻塞，
本批没有硬编码转换目标，也不宣称完整关闭 `GAP-OPP-003`。

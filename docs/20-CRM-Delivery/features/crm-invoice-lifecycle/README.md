# CRM-FEATURE-027：合同发票完整生命周期

完成日期：2026-07-14。决策依据：ADR-009、GAP-FIN-001/002、REQ-PAY-003。

## 交付结论

CRM 已具备可维护的合同发票业务台账，不再以空测试模板代替实现。交付覆盖草稿、正式开具、
部分/全部红冲、蓝票/红票作废、金额汇总、对象权限、操作轨迹、导出和前端操作闭环。

该聚合明确是 CRM 合同开票台账，不声明已完成税务申报、会计入账或外部税控平台开票。

## 领域模型

- `crm_invoice`：蓝票、红票使用同一聚合表，`direction` 区分方向；红票通过
  `original_invoice_id` 引用原蓝票，原票不会被覆盖。
- `crm_invoice_action_record`：记录创建、修改、开具、作废、红冲、红票作废和删除草稿轨迹。
- 内部申请号 `no` 与税务发票号 `invoice_no` 分离；税务发票号按租户唯一。
- 合同、客户、负责人和购方信息在票据侧保存业务快照；请求不能修改合同客户和归属。
- 数据库 CHECK 约束方向、状态、类型、正金额、累计红冲金额、红蓝票引用关系和正式票据字段。

金额公式：

```text
合同净开票额 = 有效蓝票金额 - 有效红票金额
合同剩余可开票额 = 合同金额 - 合同净开票额
蓝票剩余可红冲额 = 蓝票金额 - 蓝票累计有效红冲金额
```

正式开具、红冲和作废统一先锁合同、再锁票据。合同锁内计算净额，避免并发开票突破合同金额。

## 生命周期和接口

| 状态 | 允许动作 |
|---|---|
| 草稿 | 修改、删除、正式开具 |
| 已开具蓝票 | 作废、部分/全部红冲 |
| 部分红冲蓝票 | 继续红冲；通过作废有效红票恢复额度 |
| 已全部红冲蓝票 | 只读；可作废对应红票后恢复状态 |
| 已开具红票 | 作废，恢复原蓝票累计红冲额和状态 |
| 已作废 | 只读，不计入有效金额 |

管理端接口位于 `/crm/invoice`：

- `create`、`update`、`delete`：草稿维护；
- `issue`：正式开具；
- `red-flush`：创建独立红票；
- `void`：作废蓝票或红票；
- `get`、`page`、`contract-summary`、`export-excel`：详情、筛选、汇总和导出。

权限拆分为 query/create/update/delete/issue/red-flush/void/export，数据范围继续复用 CRM 对象权限。

## 外部适配与配置

`CrmInvoiceProvider` 冻结开具、红冲和作废三个适配命令。每个命令携带稳定幂等键，适配器必须
原样确认请求号；返回空 provider 或不同请求号会使事务回滚。

当前 YAML 显式配置：

```yaml
mitedtsm:
  crm:
    invoice:
      provider: local-ledger
```

`local-ledger` 只确认 CRM 内部台账，不生成虚假的外部票据 ID。未来接税务平台时必须维持幂等
调用和失败重试语义，不能直接写 CRM 表。

## 前端

`Web/src/views/crm/invoice/` 独立维护列表、草稿表单、开票、红冲、作废和详情轨迹组件：

- 按申请号、税务号、状态、方向、类型、客户和日期筛选；
- 草稿表单展示合同金额、有效蓝票、有效红票、净额和剩余额度；
- 专票动态要求完整购方税务信息；
- 正式票据只读，动作按钮由状态机统一函数控制；
- 详情展示外部幂等请求号和不可变生命周期轨迹；
- 开票、红冲命令统一提交 Epoch 毫秒，和 Server 全局 `LocalDateTime` JSON 契约一致；
- 中文、英文、阿拉伯文文本同步维护。

## 明确未宣称范围

- 税务平台认证、异步回调、电子票文件下载；
- 会计凭证、税务申报、进项抵扣；
- ERP 应收核销、银行到账对账；
- 发票审批流。若业务要求开票申请审批，应先新增 ADR，不能把“草稿”直接解释为“审批中”。

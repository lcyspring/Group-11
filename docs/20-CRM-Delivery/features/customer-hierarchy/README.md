# CRM-FEATURE-007：客户上下级关系

## 需求证据

- `docs/03-Gap-Analysis/03-Business-Domain-Gaps.md` 的 `GAP-CUS-002` 明确指出客户上下级关系未完善。
- `docs/02-Prototype-Analysis/user-stories/01-Core-Domains.md` 的 `US-CUS-005` 要求维护集团/分支机构关系，禁止自关联和循环，并可从客户详情进入下级列表。

## 完成范围

- `crm_customer.parent_customer_id` 保存可空的上级客户编号，增量迁移可重复执行，并建立 `(tenant_id, parent_customer_id, deleted)` 索引。
- 创建和编辑表单可选择当前用户可见的上级客户；编辑时排除客户自身。
- 客户详情和列表批量回显上级客户名称，不产生 N+1 查询。
- 详情可进入带 `parentCustomerId` 条件的客户分页，分页继续复用既有 CRM 数据权限条件，只显示当前用户有权查看的直接下级。
- 支持显式解除上下级关系；有有效下级客户时禁止删除父客户。
- 线索转换、导入等未提供层级语义的入口默认创建根客户，不推测集团关系。

## 一致性与并发模型

所有创建子客户、更新层级和删除客户的事务，先按客户 ID 顺序锁定当前租户的有效客户层级快照，再校验和写入：

1. 父客户必须存在于当前租户快照；跨租户 ID 与不存在 ID 一样拒绝。
2. 父客户不能等于自身。
3. 从候选父客户向上遍历完整祖先链，任意深度回到当前客户都拒绝。
4. 同租户层级写操作在同一锁序下串行，因此并发 `A→B` 与 `B→A` 不能分别在旧快照上同时通过。
5. 删除在相同锁内检查直接下级，避免删除和新建下级并发造成孤儿关系。

该方案优先保证层级一致性；当前租户客户量增长到锁竞争明显时，应把串行点演进为专用租户层级版本/锁记录，而不能牺牲环路校验。

## 错误码

- `1020006016`：上级客户不能是客户自身。
- `1020006017`：上级客户不存在或不属于当前租户。
- `1020006018`：客户上下级关系形成循环。
- `1020006011`：删除客户失败，有关联下级客户。

## 完成证据

- CRM `Crm*Test` 全量 50/50。
- 真实 API 覆盖创建、回显、直接下级列表、自关联、跨租户、两/三节点环、删除保护、解除关系和删除。
- 真实并发 `A→B / B→A` 一成功一拒绝，递归 SQL 检查环数量为 0。
- 幂等迁移连续执行两次通过；字段和复合索引元数据正确。
- ESLint、Web `build:prod`、Server JAR 及 rootless Podman Server/Web/Mall 通过。

详细结果与覆盖率见 `docs/20-CRM-Delivery/testing/customer-hierarchy/`。

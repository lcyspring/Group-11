# CRM 公海并发领取实现说明

完成日期：2026-07-14

## 需求与差异依据

- `US-CUS-007` 明确要求同一公海客户被并发领取时只有一人成功。
- `GAP-CUS-001` 要求补齐客户、公海、归属与协作流程的业务完整性。
- `PR-CUS-003` 将公海领取和分配纳入客户归属核心流程。
- 现有前端已经提供领取、批量领取、分配和错误消息展示，本项无需重复实现页面。

## 原问题

原 `receiveCustomer` 先执行普通 `SELECT`，再校验负责人并更新客户。两个事务可以同时读到
`owner_user_id = NULL`，随后都完成负责人权限和归属历史写入。最终客户只有一个负责人，但可能残留多个
有效 Owner 权限和多条领取历史，两个调用方也都会收到成功响应。

## 实现

- `CrmCustomerMapper.selectByIdsForUpdate` 使用 `SELECT ... FOR UPDATE` 锁定待领取客户。
- SQL 强制 `ORDER BY id ASC`，使相反顺序的批量请求也按同一主键顺序获取行锁，降低死锁风险。
- 负责人校验改在获得锁后的最新记录上执行；后到事务会读到已更新的负责人并返回
  `CUSTOMER_OWNER_EXISTS`（`1020006001`）。
- 领取、客户负责人更新、Owner 权限和 `TAKE_POOL` 归属历史仍位于同一事务。

## 前端结论

前端领取入口、批量参数、成功刷新和后端错误消息展示均已存在。真实并发中失败方可直接看到“客户已存在所属负责人”，
没有新增前端代码，避免引入一套只在浏览器内生效、无法替代数据库并发控制的重复逻辑。

## 边界

- 行锁只保护数据库事务内的最终一致性，不保证两个并发请求中由哪位用户胜出。
- 批次采用全有或全无语义：任一客户已被领取时，整个后到批次回滚并返回失败。
- 分配接口复用相同 Service 方法，因此也获得相同的串行化保护。

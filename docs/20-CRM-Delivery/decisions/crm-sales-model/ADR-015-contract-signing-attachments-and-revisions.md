# ADR-015：合同签署、附件与不可变版本

状态：Accepted（已实现）。日期：2026-07-14，2026-07-16 完成补充协议闭环。关联：ADR-002/004/006、GAP-CTR-001。

## 决策

继续以现有 `crm_contract` 作为唯一销售协议聚合，不创建重复的 CRM 销售订单。合同审批通过、
合同实际签署和合同生效是三个不同事实，不能继续用 `audit_status` 或现有 `signUserId` 推断已签署。

现有 `signUserId`、`signContactId` 表示拟定的我方签约人和客户联系人。实际签署使用独立命令，
保存签署状态、方式、时间、签署副本、稳定请求号和经办人。

## 状态与规则

1. 草稿、驳回、取消状态仍通过现有修订重提闭环编辑；
2. 只有审批通过合同可以执行签署；签约联系人必须属于合同客户；
3. 签署副本是不可变附件，签署后不能被普通附件删除接口移除；
4. 重复提交相同请求号必须幂等，不同内容不能覆盖已签署事实；
5. 已签署合同的业务字段不得原地改写。正式变更创建新版本/补充协议记录，保留前后快照和原因；
6. 作废签署必须是独立动作和轨迹，不以删除记录表达。

## 附件模型

附件使用规范化子表，不把任意 URL 数组直接塞入合同主表。每条记录至少保存合同、版本、文件
标识或 URL、原始文件名、内容类型、大小、SHA-256、类别、上传人和时间。下载、查询、删除均先
校验 CRM 合同对象权限；签署副本和已经进入正式版本的附件不可变。

## 版本与轨迹

合同变更记录保存递增版本号、动作类型、前后快照、产品行快照、操作人、原因和时间。审批提交、
审批终态、签署、作废和正式变更均进入同一不可变轨迹。列表展示摘要，详情按需加载快照，避免把
大 JSON 复制进合同分页。

正式变更使用独立 `crm_contract_amendment` 聚合。它保存基线/目标版本、前后合同与产品快照、
金额差额、幂等请求和独立流程实例；创建、修改和提交只形成候选轨迹，不推进当前有效版本。
只有 `crm-contract-amendment-audit` 审批通过后，才在同一事务中更新合同/产品投影、锁定依据附件
并写入生效轨迹。目标金额不得低于既有回款、计划和净开票事实。

## 外部签署适配

使用 `CrmContractSignProvider` 隔离电子签平台，配置键为：

```yaml
mitedtsm:
  crm:
    contract-sign:
      provider: local-record
```

命令行仍只指定 YAML 路径。Provider 必须显式返回支持的签署方式，服务端强制校验，前端只展示
能力集合。`local-record` 仅支持线下签署事实登记，不接受“电子签”命令，也不生成虚假电子签章 ID。
未来适配电子签平台时必须复用稳定幂等键和回调验签，Controller 不得直接访问供应商 SDK。

## 明确不接受

- 用 `signed=true` 一个字段代替签署证据、附件和轨迹；
- 允许审批通过或已签署合同继续走普通 update 覆盖历史；
- 只在前端过滤联系人或附件，服务端不校验对象关系和权限；
- 把基础设施文件表直接当成 CRM 合同附件权限模型。

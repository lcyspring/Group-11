# ADR-016：CRM 营销活动、客户关怀与合规群发

状态：已接受
分支：`develop`
日期：2026-07-16

## 决策

营销域采用 CRM 自有业务状态和不可变收件人快照，复用系统短信、邮件发送底座，但不把系统发送日志当作营销业务真源。

- 活动状态：草稿、进行中、已锁定、已终止、已完成；锁定后禁止新增关联对象。
- 触达渠道：短信、邮件或双渠道。短信使用联系人/客户手机号，邮件使用联系人/客户邮箱；无效地址直接排除并保留排除原因。
- 同意与退订：按租户、客户/联系人、渠道保存最新同意状态；退订优先级高于同意，任何群发前再次检查。
- 群发审核：创建者提交后进入待审核；审核人不能是创建者。通过后才允许发送，驳回可修订后重新提交。
- 频控：按 YAML 配置限制批次大小、单联系人每日次数和租户月度收件人数；达到任一限制时拒绝发送，不静默截断名单。
- 幂等与重试：每个收件人快照唯一；发送任务使用状态条件更新，成功收件人不可重复发送，失败收件人可单独重试。
- 默认提供商模式为 `record-only`，只写营销结果和系统审计；配置为 `system` 时调用现有短信/邮件 API。未配置模板或渠道账号时不得伪造成功。
- 客户数据范围复用 CRM 客户/联系人权限；群发快照建立后权限变化不影响已生成的审计记录，但发送前仍需检查对象是否删除。
- 自动关怀只生成待发送记录，必须经过同意、退订和频控检查；生日/节假日重复触达按规则键幂等。

## 配置

所有阈值置于 `mitedtsm.crm.marketing` YAML 节点，命令行不接受业务参数：

```yaml
mitedtsm:
  crm:
    marketing:
      provider-mode: record-only
      batch-size: 50
      max-batch-size: 500
      per-recipient-daily-limit: 1
      monthly-recipient-limit: 10000
      care-cron: "0 0 9 * * ?"
      care-zone: Asia/Shanghai
      lock-key: crm:marketing:care
      lock-lease-seconds: 300
```

## 不采用的方案

- 不直接从前端提交的完整手机号/邮箱列表发送；前端只能提交 CRM 对象编号。
- 不把“调用发送 API 成功”直接当作送达成功；营销结果记录发送日志编号和后续状态。
- 不用硬编码的管理员角色、配额或自动祝福时间替代配置和权限服务。

# ADR-018：客户四态生命周期与成交字段兼容

状态：Accepted。日期：2026-07-14。分支：`develop`。

## 决策

客户业务状态固定为四个稳定编码：10 潜在客户、20 意向客户、30 成交客户、40 流失客户。
状态只能通过独立命令变更，普通客户资料编辑不能修改状态。每次成功变更必须与客户主表更新在
同一事务写入不可变历史，历史保存前后状态、原因、操作人和业务时间。

转为流失客户必须填写原因；离开流失状态时清空主表当前流失原因，但历史原因永久保留。重复变更
到当前状态返回明确业务错误，不制造无效历史。状态命令先锁定客户行，串行化并发操作。

## 兼容规则

保留 `deal_status` 作为兼容派生字段，不再作为完整客户状态源：

- 生命周期为 30 时，`deal_status=true`；
- 生命周期为 10、20、40 时，`deal_status=false`；
- 旧成交接口继续存在，`true` 映射为 30，`false` 映射为 10，并同样写入历史；
- 公海与客户限额规则暂时继续读取兼容字段，避免迁移同时改变回收语义。

旧数据迁移中，已成交映射为 30，其他客户映射为 10。不从缺失数据推测意向或流失状态。

## 统计与读取

客户分页支持按四态筛选，详情返回当前状态、变更时间和当前流失原因。原“客户状态分析”接口保持
URL 不变，但响应和 SQL 改为按四态分组，前端饼图和表格统一读取生命周期状态。

## 数据约束

客户表增加状态、状态变更时间、当前流失原因和组合索引，并用 CHECK 锁定四态值、成交兼容字段
同步及流失原因一致性。历史表对前后状态设置 CHECK，对流失原因设置条件 CHECK。迁移可重复执行，
并同时接入新库初始化和已有 Podman 数据卷兼容清单。

## 后续边界

本 ADR 不定义公海对意向/流失客户的差异化回收策略。该策略继续沿用兼容成交字段，待公共池、
保护期和回收规则统一决策后再升级，不在本次状态模型中硬编码业务阈值。

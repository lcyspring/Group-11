# ADR-013：组织范围与 CRM 对象权限合并

状态：Accepted（MVP）

日期：2026-07-15

实施分支：`develop`

## 合并算法

一次 CRM 请求按以下顺序校验，任一步拒绝即停止：

1. 租户插件限定当前租户；
2. `@PreAuthorize` 校验功能权限字符；
3. `CrmAuthorizationService` 合并直接对象授权和组织读范围；
4. 业务服务校验状态、金额、关联关系、幂等和敏感字段；
5. 附件下载继续继承父 CRM 对象权限。

系统角色数据范围由平台 API 统一解析：

| 平台范围 | CRM 负责人范围 |
|---|---|
| ALL | 当前租户全部负责人 |
| DEPT_CUSTOM | 自定义部门内有效用户 |
| DEPT_ONLY | 本部门内有效用户 |
| DEPT_AND_CHILD | 本部门及子部门内有效用户 |
| SELF | 当前用户 |
| API 异常返回空 | 失败收敛到当前用户，不开放全量 |

## 并集与交集规则

- READ = 当前用户直接 READ/WRITE/OWNER 对象，并集负责人位于组织范围的对象；
- WRITE/OWNER = 只认当前用户直接对象权限，组织范围不参与；
- 多角色的功能权限和平台部门范围由系统模块先取并集；
- “下属负责”先得到下属集合，再与组织可读负责人集合取交集；
- 流程候选人/受理人必须另有对象可读权限，不能作为隐式对象授权；
- 导出逐对象要求当前用户直接 WRITE/OWNER，不复用组织 READ；
- 公海对象继续按既有“无负责人可读、领取后受对象权限控制”规则处理。

## 列表场景

CRM 通用列表场景固定为：`1 我负责`、`2 我参与`、`3 下属负责`、`4 组织范围`。组织范围已接入
线索、客户、联系人、商机、合同、回款计划、回款、退款和发票。管理员选择组织范围时查看当前租户全量；
普通用户根据 YAML 指定的管理员角色例外和系统角色数据范围计算。

## 不接受的实现

- 直接把通用 `dept_id` SQL 规则套到只有 `owner_user_id` 的 CRM 主表；
- 仅凭上下级关系自动获得修改或导出能力；
- 在各 Controller 重复判断角色名称；
- 把流程候选人、工单处理组或附件 URL 当作独立数据权限来源；
- 组织权限 API 异常时回退到全部数据。

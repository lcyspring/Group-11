# STAT-BUG-018：成交周期区间聚合累加每日平均值

## 基本信息

- 发现日期：2026-07-13
- 级别：P1
- 模块：CRM / 数据统计 / 客户成交周期
- 状态：已关闭
- 影响接口：按日期成交周期（周、月、季度、年间隔）

## 现象与影响

Mapper 先按天计算平均成交周期，Service 在用户选择周、月、季度或年时，把区间内每天的平均值直接求和。这既不是整体平均，也不是加权平均；区间越长，结果通常越大。

例如三个客户周期分别为 5、20、30 天，分布在三个成交时间点，旧逻辑会得到 55，正确整体平均是 18.3。

## 根因

日期维度 SQL 提前丢失客户级明细和每日样本量，Service 只能拿到日均值。合并区间时错误复用了适用于“次数”的 `sum()` 操作。

## 修复关键

- Mapper 返回每个客户首次合同的成交时间和周期，不再提前按天平均。
- 时间字符串固定输出 ISO `yyyy-MM-ddTHH:mm:ss`，与公共日期解析器一致。
- Service 在每个请求区间内对客户级周期调用 `average()`。
- 结果按既有规则向零方向截断到一位小数。
- 新增 Service 单元测试，覆盖同一月不同日期、同日多个客户的非均匀样本。

## 验证证据

1. 新增测试样本 5、20、30，月维度断言 18.3；首次运行发现空格时间不兼容并在提交前修正为 ISO 格式。
2. CRM 统计专项测试 7/7 通过。
3. Server 生产 JAR 构建通过，最新产物重建 rootless Podman 后服务健康。
4. Podman 创建三名临时客户及首次审批合同，周期分别为 5、20、30 天。
5. 月间隔 API 返回 `time=2026-07, customerDealCycle=18.3`。
6. 三名临时客户和三份合同均已物理清理，两表残留为 0。

## 遗留风险

本修复纠正区间数学算法，不处理 `STAT-DATA-002` 的负周期口径。负数仍参与真实平均，直到产品明确历史补录规则。

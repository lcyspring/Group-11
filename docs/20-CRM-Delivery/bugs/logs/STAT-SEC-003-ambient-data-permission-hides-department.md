# STAT-SEC-003：统计范围校验被当前用户数据权限二次过滤

发现/关闭日期：2026-07-16。级别：P0 / 权限可用性与安全边界。

## 现象

真实创建仅有客户统计权限、数据范围为 SELF 的临时用户后，该用户查询自己的统计仍返回
`1020014006`。用户、角色、部门和 `data_scope=5` 在数据库中均正确，既有 Mockito 测试未暴露问题。

## 根因

`CrmStatisticsDataScopeService` 为了验证不可信的 `deptId/userId`，会调用组织 API 读取部门和部门
人员。但该读取继承了当前登录用户的系统数据权限；SELF 规则对 `system_dept` 生成了无效的
`null = null` 条件，导致真实部门被过滤成不存在，统计范围统一失败关闭。

## 修复关键

在统计范围验证入口显式使用 `@DataPermission(enable = false)`，只在组织解析这一小段关闭环境数据
权限。随后仍由 CRM 自己的完整校验执行：部门存在、人员属于部门树、ALL/SELF/部门范围覆盖和
失败关闭。统计 Mapper 查询本身不受该注解影响，仍按已验证的 `userIds` 执行。

拒绝分支增加结构化日志，记录登录人、目标部门/人员和解析后的只读范围，便于后续权限排障。

## 验证

- 注解契约回归和既有统计范围专项全部通过；
- Ubuntu 26.04 CRM 自动化 427/427；
- 真实临时 SELF 用户查询本人客户统计成功，API 与 MySQL 均为 1；
- 查询同部门其他人员和整部门返回 `1020014006`；
- 仅授予客户统计权限时访问漏斗目录返回 403；
- 跨租户返回 403，无令牌返回 401；
- 临时客户、用户、角色、授权和令牌均自动清理。

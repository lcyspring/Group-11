# CRM-MKT-BUG-004：群发“已发送”被误当成送达结果

日期：2026-07-17。分支：`develop`。

## 现象

群发任务只显示已发送和失败。系统短信、邮件 API 返回日志编号后，收件人立即进入 `SENT`，但短信
回执、邮件异步 SMTP 结果均未回收，页面无法区分短信已送达、短信仍在处理中、邮件服务商接受和
邮件首次打开。原型要求的到达率、打开率因此没有真实数据基础。

## 根因

- 收件人表缺少独立于发送生命周期的提供商结果字段；
- CRM 没有读取 `system_sms_log` 回执和 `system_mail_log` 异步结果的内部契约；
- 邮件正文没有不可猜测的打开追踪令牌；
- 调度任务只发送定时群发，不回收已创建日志的后续状态；
- 前端把任务发送状态和最终到达结果放在同一口径中。

## 修复

- 新增 `delivery_status/delivered_at/opened_at/tracking_token` 和幂等迁移；
- 短信按发送状态与接收回执计算送达，邮件只把 SMTP 成功标记为“服务商接受”；
- 系统邮件仅在 CRM 传入保留参数时追加经过 HTTP(S) 校验和 HTML 转义的 1×1 像素；
- 匿名像素端点对有效、无效令牌返回相同图片，首次打开幂等写入且不返回业务数据；
- 调度和手工刷新共同回收提供商结果；
- 页面独立展示短信送达率、邮件接受数、打开数、打开率和逐收件人时间。

## 边界

`record-only` 只表示合规留痕，不制造送达或打开数据。邮件 SMTP 接受也不等同于最终投递；当前
邮件能力明确展示“服务商接受”和“首次打开”。真实外部邮件必须把 YAML 中公开基地址配置为邮件
客户端可访问的 HTTPS 地址。

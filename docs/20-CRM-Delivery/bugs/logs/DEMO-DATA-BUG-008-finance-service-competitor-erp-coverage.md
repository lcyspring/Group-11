# DEMO-DATA-BUG-008：财务负责人、客服处理组、竞争对手和 ERP 映射缺失

状态：Fixed

日期：2026-07-19

分支：`develop`

## 现象

- 管理员登录后，退款/冲销与报销管理的“我负责的”基本为空；
- 客服处理组、竞争对手、ERP 客户映射和 ERP 产品映射页面没有数据；
- 工单虽然有处理人，但没有关联处理组，无法验证分组派单链路。

## 根因

数据集只把退款与报销负责人分配给两名演示财务人员，没有为 KDL 中的基准管理员生成 OWNER 样本；
cleanup 已覆盖处理组、竞争对手和 ERP 映射相关数据，但生成 SQL 从未创建这些对象。此前校验只验证
财务状态与总行数，没有验证负责人场景、客服组成员关系或 ERP 映射引用完整性。

## 修复关键

- KDL 新增竞争对手、客服处理组、ERP 客户映射、ERP 产品映射四个显式数量字段；
- 退款和报销负责人按基准管理员及两名财务人员分布，同时为非 OWNER 数据补齐管理员 INVOLVED 关系；
- 生成 4 个客服处理组、12 条成员关系，并让 160 条工单全部关联有效处理组；
- 生成 50 条竞争对手资料，覆盖多个负责人和启用状态；
- 分别生成 50 条 ERP 客户、ERP 产品主数据及一对一 CRM 映射；
- replacement cleanup 只识别带生成批次标记的 ERP 主数据，不删除人工维护的 ERP 数据；
- 校验 SQL 新增负责人场景、处理组成员、竞争对手数量和 ERP 双向引用断言。

## 职责边界

生成器仍只生成 SQL、manifest 和 checksum，不连接数据库。实际旧数据清理和新数据导入由
`deploy.sh` 根据运行 KDL 的 `mysql.dataset_mode "replace"` 在部署阶段完成。

## 回归

- 数据生成器契约测试通过，生成文件 checksum 全部通过；
- `deploy.sh` replace 数据集自校验通过：退款 30、报销 60、竞争对手 50、处理组 4、两类 ERP 映射各 50；
- 管理员 API：OWNER 退款 10、OWNER 报销 20，INVOLVED 退款 20、INVOLVED 报销 40；
- 客服处理组 API 返回 4 组、12 个成员关系；竞争对手和两类 ERP 映射分页总数分别为 50、50、50；
- 8080、8081、8082 服务均已恢复并通过部署健康检查。

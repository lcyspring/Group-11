# CRM-LEAD-001：线索并发转换可重复创建客户

发现/关闭日期：2026-07-13
级别：P0
关联差异：`GAP-LEAD-002`

## 现象与风险

原转换流程先读取 `transform_status`，确认未转换后创建客户，最后才更新线索。两个并发请求可以同时读取到未转换状态并各自创建客户，导致同一线索关联结果不确定，并产生孤立或重复客户。简单的前置查询不能提供幂等保证。

## 根因

“检查状态”和“声明转换权”不是同一条原子数据库操作；事务隔离只能保护各自事务内的一致性，不能让两个请求互斥通过应用层判断。

## 修复

- Mapper 使用 `id + transform_status=false` 条件更新，转换事务先把状态原子更新为 true。
- 只有影响一行的请求继续创建客户；影响 0 行统一返回 `1020001001`。
- 客户、首联系人、线索客户 ID 回写和跟进复制保持在同一个事务，后续任一步失败都会回滚抢占状态。

## 验证

- Service 单元测试验证抢占失败后不会调用客户创建，并验证成功调用顺序为 `claim → customer → contact → link`。
- 真实 MySQL 两个并发 API 请求仅一个成功，另一个返回 `1020001001`；最终有效客户和联系人均恰好一条。
- 受控联系人 INSERT 失败后，线索 `transform_status=0`、`customer_id=NULL`，客户和联系人计数均为 0。
- 重复请求验证客户/联系人仍为 1/1；CRM 全量 39/39，联系人回归 18/18。
- 临时线索、客户、联系人、日志、权限和故障触发器最终均为 0。

## 状态

已关闭。

# CRM-INV-BUG-003：格式化开票日期被静默写成 Unix 纪元

发现日期：2026-07-14。状态：Fixed。

## 现象

真实 API 生命周期验收以 `2026-07-14T19:20:00` 调用正式开票、以相同格式调用红冲，接口均返回
成功，但详情响应的 `invoiceDate` 为 `0`，MySQL 中为 `1970-01-01 08:00:00`。

## 根因

Server 全局 `LocalDateTime` JSON 契约是 Unix Epoch 毫秒。发票开票和红冲表单却使用 Element Plus
的 `value-format="YYYY-MM-DD HH:mm:ss"`，TypeScript 也把命令日期声明成普通字符串。全局反序列化器
对非数值字符串调用 `getValueAsLong()` 后得到 `0`，因此请求未失败，而是静默写入错误日期。

## 修复

- 开票、红冲日期选择器改用 `value-format="x"`；
- 提交前统一由 `toInvoiceEpochMillis` 校验并转换为正的整数毫秒值；
- API 命令、响应和轨迹时间类型明确为 `number`，与 Server 全局契约一致；
- 详情展示显式从 Epoch 毫秒构造 `Date`；
- 新增前端回归测试，拒绝 `YYYY-MM-DD HH:mm:ss` 这类格式化命令日期。

没有为发票引入字段级特殊日期格式，避免同一系统同时存在两套命令时间契约。

## 验证

- Ubuntu 26.04 前端专项测试：3/3；
- 发票目录 ESLint：0 警告；
- Ubuntu 26.04 Web 生产构建：通过；
- API 使用 `1784028120000` 红冲后，详情返回同一毫秒值，MySQL 落库为
  `2026-07-14 19:22:00`。

## 数据影响

缺陷在新发票模块尚未提交、未推送时由本机验收发现。当前无已发布生产数据需要迁移。本机验收中
产生的两条错误日期样本使用精确税务票号进行一次性修正，并保留独立运行记录；不得编写无条件更新
全部 `1970-01-01` 数据的通用迁移。

# CRM-CORE-BUG-013：公海客户可被多人并发领取

- 发现日期：2026-07-13
- 修复日期：2026-07-14
- 模块：CRM / 客户公海
- 严重级别：高
- 状态：已修复并验证
- 需求关联：`US-CUS-007`、`GAP-CUS-001`、`PR-CUS-003`

## 现象

用户 1 和用户 100 同时领取公海客户 `740005`，两个 API 请求均返回 code 0。数据库最终出现：

- 客户最终负责人为用户 100；
- 用户 1 和用户 100 各有一条有效 Owner 权限；
- 用户 1 和用户 100 各写入一条 `TAKE_POOL` 归属历史。

这会造成客户负责人字段、权限和审计历史互相矛盾。

## 根因

`CrmCustomerServiceImpl.receiveCustomer` 使用普通 `selectByIds` 读取客户。负责人为空的检查与随后更新之间没有行锁或条件更新，
两个事务可以同时通过检查并分别写入权限与历史，属于典型的检查后执行竞态（TOCTOU）。

## 修复

- 新增 `CrmCustomerMapper.selectByIdsForUpdate`。
- 在领取/分配事务中按 ID 升序执行 `SELECT ... FOR UPDATE`。
- 后到事务获得锁后读取最新负责人，使用既有错误码 `CUSTOMER_OWNER_EXISTS`（`1020006001`）拒绝请求。
- 保持负责人、权限和归属历史在同一事务中提交或回滚。

## 验证

| 场景 | 修复前 | 修复后 |
|---|---|---|
| 单客户双用户并发 | 两个请求都成功 | 一个成功，一个 `1020006001` |
| 有效 Owner 权限 | 两条 | 一条 |
| `TAKE_POOL` 历史 | 两条 | 一条 |
| 相反顺序批量并发 | 未保证锁顺序 | 无死锁，一个批次成功 |
| CRM 全量测试 | 既有基线 | 56/56 |

## 数据处理

修复前复现客户 `740005` 已软删除，有效客户和权限为 0；修复后测试客户 `740006` 至 `740008` 已物理清理。

## 遗留风险

当前依靠 MySQL/InnoDB 事务行锁，若未来将客户写入拆到异步任务、跨库或非事务存储，需要重新设计分布式并发控制，不能假定本行锁仍覆盖完整流程。

# STAT-BUG-017：成交周期使用客户全部合同

## 基本信息

- 发现日期：2026-07-13
- 级别：P1
- 模块：CRM / 数据统计 / 客户成交周期
- 状态：已关闭
- 影响接口：按日期、员工、区域、产品的四个成交周期接口

## 现象与影响

服务接口把成交周期定义为“客户创建到合同第一次成交的时间差”，但四个 SQL 都连接客户的全部审批合同。同一客户有多份合同时会被重复计入，员工和区域平均值被后续合同拉长，日期维度产生多个成交日，产品维度还会混入后续合同产品。

既有客户有两份审批合同，旧员工/区域平均周期为 `-12`，实际首次合同周期为 `-16`；日期维度同时出现 `-16` 和 `-8` 两天。

## 根因

“首次合同”只写在 Service/Controller 注释中，没有落实为 Mapper 的公共选择条件。各维度独立复制了普通合同连接，因此语义逐步漂移。

## 修复关键

- 新增 MyBatis SQL 片段 `firstApprovedContractJoin`。
- 每个客户按 `order_date` 升序选择最早审批通过合同，同日按合同 ID 稳定选择，空日期排最后。
- 首次合同明细与四个成交周期查询共同复用该片段。
- 日期、员工、区域和产品维度不再各自维护不同合同筛选逻辑。

## 验证证据

1. Maven CRM 统计专项测试 6/6 通过，Server 生产 JAR 构建通过。
2. 最新 JAR 重建 rootless Podman，Server/Web/Mall 健康。
3. 既有双合同客户：
   - 员工/区域周期由全合同平均 `-12` 改为首次合同 `-16`。
   - 日期维度只保留首次合同日期 2024-02-07，不再出现第二合同日期 2024-02-15。
4. 构造正向临时数据：客户创建于 2026-07-01，审批合同分别下单于第 5 天和第 20 天，并关联不同产品。
5. 四类 API 对账：
   - 日期：仅 2026-07-06，周期 5。
   - 员工：周期 5，客户数 1。
   - 区域：北京市，周期 5，客户数 1。
   - 产品：仅首次合同的“商品002”，周期 5，客户数 1。
6. 临时客户、2 份合同、2 条合同产品已物理清理，三表残留均为 0。

## 遗留风险

周期仍可能因历史下单日期早于客户创建日期而为负；该问题单列 `STAT-DATA-002`，本修复不通过取绝对值或归零掩盖数据异常。

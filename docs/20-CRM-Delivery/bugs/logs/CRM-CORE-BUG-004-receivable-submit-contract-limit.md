# CRM-CORE-BUG-004：回款草稿提交时未重新校验合同额度

## 基本信息

- 发现日期：2026-07-13
- 级别：P0
- 模块：CRM / 回款管理 / 审批提交
- 状态：已关闭
- 影响接口：`PUT /admin-api/crm/receivable/submit`

## 现象与影响

回款草稿在创建或编辑时会校验合同剩余额度，但提交审批时只校验草稿状态。用户可以先为同一合同创建多张未占额度的草稿，再依次或并发提交，导致“审批中 + 审批通过”的回款总额超过合同金额，破坏财务金额守恒。

现有历史数据也存在合同金额与回款状态合计不一致的样本。历史异常只作为数据问题报告，本次不自动改账。

## 根因

- 草稿不参与合同额度占用。
- `submitReceivable` 没有在草稿转为审批中之前重新计算剩余额度。
- 同合同提交之间没有数据库串行化措施，两个事务可以同时通过相同的余额判断。
- 同一草稿的并发请求可能在等待锁前读到相同旧状态，因此锁后还必须重新读取。

## 修复关键

- 在回款提交事务中对所属 `crm_contract` 执行 `SELECT ... FOR UPDATE`。
- 获得合同锁后重新读取回款，重新确认它仍为草稿，阻止同一草稿重复提交。
- 按最新的审批中、审批通过回款重新校验合同剩余额度；当前草稿自身被排除。
- 仅当额度校验通过后才创建 BPM 流程并把回款改为审批中。

## 验证证据

1. 新增 `CrmReceivableServiceImplTest`：合同金额 100、已有审批中回款 60、待提交草稿 50。
2. 单元测试确认先锁定合同、锁后第二次读取草稿；剩余额度 40 时抛业务错误，BPM API 未被调用。专项测试 1/1 通过。
3. Server 生产 JAR 构建通过，并替换现有 rootless Podman 后端容器；健康检查返回 `UP`。
4. Podman 真实库构造同样的 100/60/50 场景，提交接口返回业务码 `1020004007` 和剩余金额 `40.000000`。
5. 提交失败后草稿仍为状态 0、`process_instance_id` 仍为空，证明没有进入审批流程。
6. 临时客户、合同和两张回款记录均已物理清理，三类残留计数均为 0。

## 遗留风险

- 历史超额合同需要业务负责人确认后人工调账或迁移，本修复不推断真实财务状态。
- `selectReceivablePriceMapByContractId` 的展示口径问题已由 `STAT-BUG-028` 修复为只累计审批通过回款。
- 后续仍需增加两个真实并发 HTTP 请求的压力型回归；本次数据库锁顺序由单元测试和真实额度拒绝场景覆盖。

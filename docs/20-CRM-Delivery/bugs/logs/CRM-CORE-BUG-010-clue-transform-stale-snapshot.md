# CRM-CORE-BUG-010：线索转换等待行锁后仍可能使用旧字段

发现/关闭日期：2026-07-13
级别：P0
关联差异：`GAP-LEAD-002`

## 现象与根因

转换先普通读取线索，再通过条件 UPDATE 抢占转换状态。如果一个更新请求已排在线索行锁队列前，转换的普通查询不会等待，会先建立 MySQL `REPEATABLE READ` 快照；之后条件 UPDATE 等待更新提交，但客户创建仍使用等待前读取的旧线索对象，导致客户名称、手机号等来源字段落后于最终线索。

## 修复

- 转换入口首先执行线索 `SELECT ... FOR UPDATE` 当前读。
- 更新、删除、转移、跟进和团队权限写操作使用同一线索行锁。
- 保留转换状态条件 UPDATE 作为防御性原子校验。

## 验证

- 转换先进入锁队列时：转换成功，后排更新返回 `1020001002`，线索与客户保持原名称。
- 更新先进入锁队列时：更新和转换依次成功，线索及客户的名称、手机号均为更新后值。
- 两组均由真实 MySQL 外部事务短时持锁构造，结果 2/2；测试数据已清理。

## 状态

已关闭。

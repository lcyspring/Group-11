# SYSTEM-NOTIFY-FE-BUG-001：通知弹层首次打开慢且需要重复点击

状态：Fixed

日期：2026-07-15

分支：`develop`

## 现象

点击管理端顶部铃铛后，弹层可能先显示空白；用户再次点击会关闭正在加载的弹层，随后需要反复点击才看到
站内信。铃铛周围的头部留白也不属于可点击区域，进一步放大了“点击无响应”的感受。

## 根因

- 列表请求绑定在内部 SVG 图标的 `click`，而弹层显隐由外层 Popover 独立处理；
- 首次打开没有预取、缓存、加载骨架、空态或失败重试；
- 快速重复点击会并发请求并反复切换弹层显隐；
- 仅查看弹层就把未读数强制清零，红点状态与服务端事实不一致；
- 组件卸载时没有释放两分钟轮询定时器，布局重建后可能累积重复轮询。

运行日志显示 `get-unread-list` 数据库和接口链路约 11 ms，瓶颈在前端交互时序而非通知查询 SQL。

## 修复关键

- 弹层改为受控显隐，由 Popover `show` 生命周期统一刷新列表；
- 登录布局挂载时预取最近十条消息，悬停和键盘聚焦也可触发无副作用预取；
- 新增最近成功缓存与并发请求合并，弹层优先显示缓存并后台刷新；
- 首次加载显示骨架，空列表和失败重试均有明确状态；
- 扩大按钮命中区域并补充 `aria-label`；
- 查看消息不再伪造已读状态，未读红点继续以服务端计数为准；
- 组件卸载时释放轮询定时器，退出登录时清除通知缓存。

## 回归

请求合并、缓存刷新、失败重试与缓存清理专项 3/3；通知目录 ESLint 零警告；Ubuntu 26.04 Web production
build 通过。真实接口仍为单次返回一条未读消息，未修改任何通知业务数据。

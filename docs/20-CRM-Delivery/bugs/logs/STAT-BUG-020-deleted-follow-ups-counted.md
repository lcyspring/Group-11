# STAT-BUG-020：逻辑删除跟进仍进入员工客户统计

更新日期：2026-07-14

## 现象

员工客户分析的五条跟进 SQL 直接读取 `crm_follow_up_record`，没有 `deleted = 0`：

- 按日期跟进次数；
- 按日期跟进客户数；
- 按员工跟进次数；
- 按员工跟进客户数；
- 按跟进方式次数。

排行榜的同源 SQL 已过滤逻辑删除，导致不同统计页面对同一数据给出不同结果。

## 根因

XML 自定义 SQL 不会自动获得 MyBatis-Plus 基于实体 Mapper 的逻辑删除条件；开发者遗漏了显式过滤。

## 修复关键

五条 SQL 均在 `WHERE` 首部增加 `deleted = 0`，不改变人员、时间与业务类型过滤。

## 验证

- Mapper XML 回归遍历五个 statement，1/1 通过。
- Ubuntu 26.04 容器 CRM 全量 61/61。
- 真实 MySQL/API 对账留在 `STAT-DATA-001` 阶段，不提前宣称完成。

## 状态

已关闭。

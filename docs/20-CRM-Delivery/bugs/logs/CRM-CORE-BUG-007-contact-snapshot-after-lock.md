# CRM-CORE-BUG-007：客户锁后仍读取联系人旧快照

发现/关闭日期：2026-07-13
级别：P0
关联差异：`GAP-CUS-002`

## 现象与风险

联系人更新和删除先普通读取联系人，再等待客户行锁。获得客户锁后代码仍通过普通 `selectById` 重读联系人。在 MySQL `REPEATABLE READ` 下，该查询沿用事务第一次读取形成的快照，不能看到等待期间已经提交的新状态。

- 普通联系人等待期间被另一请求设为首联系人后，旧表单仍可能把它改回非首联系人，最终客户没有首联系人。
- 普通联系人等待期间被设为首联系人后，删除请求仍可能按旧快照放行删除。
- 联系人等待期间被移动到其他客户后，请求已锁定的客户集合可能与最新归属不一致。

## 根因

客户行锁只串行化修改顺序，不会自动刷新事务的一致性读视图。锁后必须对联系人本身执行当前读；普通 `SELECT` 无法作为并发状态校验依据。

## 修复

- Mapper 新增 `selectByIdForUpdate`，以 `SELECT ... FOR UPDATE` 获取联系人最新有效行。
- 更新和删除在客户行锁后统一使用该当前读。
- 比较锁前快照与当前读的客户归属；归属变化时返回 `1020003009`，要求调用方重试，避免在未锁定最新客户的情况下继续写入。
- 保留原生命周期错误：最新状态已经是首联系人时，取消返回 `1020003004`，删除返回 `1020003005`。

## 验证

- `CrmContactServiceImplTest` 16/16，通过 4 个锁后状态专项用例：更新/删除读取最新首联系人状态、更新/删除拒绝客户归属变化。
- CRM `Crm*Test` 全量 33/33。
- 真实 MySQL 锁等待验证 2/2：旧表单更新返回 `1020003004`，旧快照删除返回 `1020003005`；最终联系人仍存在且客户恰好一个首联系人。
- 首联系人原有真实 API 9/9、手机号唯一真实 API 7/7 均回归通过。
- Server 生产 JAR 构建和 rootless Podman 重建通过。
- 临时客户、联系人、操作日志、权限最终均为 0。

## 状态

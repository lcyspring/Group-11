# CRM-CLUE-POOL-BUG-004：容量守卫分步加锁产生锁升级死锁

- 日期：2026-07-15
- 分支：`develop`
- 状态：已修复

## 问题

守卫实现先执行 `INSERT IGNORE`，再执行 `SELECT ... FOR UPDATE`。守卫行已经存在时，并发事务
可能先各自取得共享锁，再同时升级排他锁。容量被守住，但其中一个 HTTP 请求返回 500，日志
为 `DeadlockLoserDataAccessException`。

## 修复

改为单条 `INSERT ... ON DUPLICATE KEY UPDATE id = id`。无论守卫首次创建还是已经存在，语句
都直接取得唯一键排他锁并持有到事务结束，不再进行共享锁到排他锁的升级。

## 验证

重新构建和部署后重复 99 + 2 并发场景：一次 `code=0`，一次稳定返回上限业务码
`1020001007`；最终持有量 100，服务日志无新增死锁和 500。

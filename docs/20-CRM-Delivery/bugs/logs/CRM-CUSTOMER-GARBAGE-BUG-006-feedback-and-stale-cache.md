# CRM-CUSTOMER-GARBAGE-BUG-006：转入反馈、缓存刷新和操作列不稳定

- 日期：2026-07-16
- 分支：`develop`
- 状态：已修复

## 问题

管理员在公海把客户转入垃圾池时，若客户仍有进行中商机或未完结销售单据，后端复用了公海迁移的
错误码，提示“不能移入公海”，与当前操作不一致。转入成功后，已被管理端页签缓存的垃圾池页面只在
首次挂载时查询，重新进入该页仍显示旧列表，必须手动刷新浏览器。

垃圾池固定操作列宽度不足且按钮允许受全局相邻按钮间距影响，在不同语言或可视宽度下可能换行、裁切，
表现为“永久删除”偶尔不显示。

## 根因

- 公海和垃圾池迁移共用了 `CUSTOMER_POOL_ACTIVE_BUSINESS`、`CUSTOMER_POOL_ACTIVE_CONTRACT`；
- 垃圾池列表使用 `onMounted` 加载，Vue `KeepAlive` 再次激活组件时不会重新执行挂载钩子。
- 操作列只设置较小最小宽度，没有为双操作提供不换行布局。

## 修复

- 为垃圾池迁移增加独立的活跃商机和受保护合同错误码，错误文本明确为“不能转入垃圾池”；
- 垃圾池列表改为在组件每次 `onActivated` 时重新分页查询，不依赖硬编码路由名；成功转入同时递增共享失效
  版本，使已经缓存的垃圾池立即后台重取，并合并相邻重复请求；
- 转入成功后仍立即刷新公海列表，保证来源页与目标页分别读取服务端最新状态。
- 操作列使用固定宽度和不换行的 `inline-flex` 布局，恢复与永久删除权限指令保持不变。

## 验证

- 垃圾池 Service 专项 9/9；活跃商机和受保护合同分别验证专属错误码、完整中文提示和零状态写入；
- 缓存、失效通知和操作列回归 3/3；
- Ubuntu 26.04 CRM 433/433，失败 0、错误 0、跳过 0；Server/Web production build 通过。

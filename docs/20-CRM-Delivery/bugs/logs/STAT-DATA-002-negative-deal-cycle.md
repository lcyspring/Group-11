# STAT-DATA-002：合同下单日期早于客户创建日期

## 基本信息

- 发现日期：2026-07-13
- 级别：P1 数据质量/口径
- 模块：CRM / 合同 / 客户成交周期
- 状态：已关闭（保留真实值并显式标记）

## 现象

既有客户 16 创建时间为 2024-02-23，首次审批合同下单日期为 2024-02-07，成交周期因此为 `-16` 天。统计 SQL 如实暴露负值，但负周期不符合持续时间直觉。

## 可能原因

- 客户在合同实际下单后才补录 CRM，历史回填本身合法。
- 合同表单允许选择早于客户创建时间的下单日期，缺少业务校验。
- “成交周期”产品定义可能应使用合同录入时间或审批完成时间，而不是可回填的下单日期。

## 根因与决策

- 已有样本是先发生交易、后补录客户的历史数据，不能自动判定为合同日期错误。
- 不修改历史业务数据，不取绝对值、不静默归零、不排除样本，避免篡改原始事实和平均值。
- 成交周期继续定义为“客户创建时间到首个审批通过合同下单时间的自然日差”。
- 日期、员工、地区、产品四类响应都增加 `negativeSampleCount`，且分组查询按客户去重计数。
- 管理端三张明细表增加“数据质量”列，用警告标签展示“负样本数量”，并解释为“历史补录异常样本”。
- 成交周期图的第一 Y 轴不再强制 `min: 0`，负周期不会被图表裁掉。
- `application.yaml` 增加 `customer.deal-cycle` 指标口径，显式声明负值参与平均并通过计数字段暴露。

## 修复关键

1. SQL 在四类聚合中同时返回周期和负样本数，产品关联多行时也以客户 ID 去重。
2. Service 在日期区间合并及员工合并时显式累加负样本数，不丢失 Mapper 结果。
3. 前端 API 类型、地区映射和产品归一化函数保留新字段；员工表类型同步改为真实员工响应。
4. 中文、英文和阿拉伯文统一增加数据质量文本。
5. 新增只接收 YAML 路径的真实验收脚本，对 MySQL 和四类 API 做负样本对账。

## 验证证据

- Ubuntu 26.04 CRM 自动化：429/429，失败 0、错误 0、跳过 0。
- 统计前端：11/11；Node 覆盖率行 95.49%、分支 95.83%、函数 100%。
- 统计目录 ESLint 零警告，Server/Web production build 通过。
- 真实运行对账：用户 1 的 MySQL 负样本数为 1、周期 `-16`；日期、员工、地区、产品及元数据 API 全部通过。
- 验收未插入、重绑或删除业务数据，无测试残留。

## 遗留边界

历史负值依然保留，这是本次的明确数据治理结论，不是未修复的计算错误。若以后业务要求禁止补录或改用审批时间，必须先签署新口径，再修改显式 YAML、写入校验和历史迁移；不得只在前端把负值改成 0。

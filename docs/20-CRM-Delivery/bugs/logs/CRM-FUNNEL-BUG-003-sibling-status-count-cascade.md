# CRM-FUNNEL-BUG-003：销售漏斗同级状态数量串增

- 日期：2026-07-18
- 分支：`develop`
- 状态：已修复

## 问题

商机状态从“吃饭”变更为“阿豆豆”后，“阿豆豆”数量增加 1，但没有发生状态变更的同级状态
“睡觉”也错误增加 1。商机操作日志仅记录“吃饭 → 阿豆豆”。

## 根因

漏斗 SQL 已使用 `business.status_id = status.id` 精确聚合当前状态，状态变更也只更新一条商机记录；
错误位于统计服务层：它按状态排序从后向前累计数量和金额，把独立状态错误解释成“至少到达本阶段”。
所以一个状态的增量会传播到排序在它之前的其它状态。

漏斗明细查询同时使用 `status.sort >= 当前排序`，导致点击某一状态后也会混入其它状态的商机，
与表格数量口径不一致。

## 修复

- 保留 SQL 按 `status_type_id + status_id` 返回的独立状态数量和金额，不再反向累计；
- 漏斗每个状态仅代表当前处于该状态的商机；
- 状态明细按精确 `status_id` 查询，不再按排序范围查询；
- 保留状态组校验、人员数据范围、统计时间范围和终态分布逻辑。

## 验证

- Ubuntu 26.04 容器 CRM 全量测试：536/536 通过；
- 专项服务测试验证三个同级状态数量保持 `10 / 4 / 1`，不再变为 `15 / 5 / 1`；
- 专项 Mapper 测试验证明细 SQL 包含精确状态条件，且不含 `>=` 范围条件；
- 运行态真实状态组返回“吃饭 2、睡觉 0、阿豆豆 2”，三个详情总数也分别为 `2 / 0 / 2`；
- JaCoCo 总体行覆盖率 49.00%，本次涉及的漏斗服务行覆盖率 66.92%。

# CRM-CORE-BUG-006：联系人手机号缺少必填和客户内唯一约束

## 基本信息

- 发现日期：2026-07-13
- 级别：P0
- 模块：CRM / 联系人
- 状态：已关闭
- 影响接口：`POST /admin-api/crm/contact/create`、`PUT /admin-api/crm/contact/update`
- 对应需求：v6 CUS-009、`GAP-CUS-002`

## 现象与影响

联系人保存请求中的 `mobile` 只有格式注解。现有校验器明确对空值放行，前端也没有必填规则；Service 和数据库均不检查同一客户下的重复手机号。因此同一客户可以保存多个空手机号联系人或多个相同手机号联系人，联系人识别、首联系人展示和后续沟通记录都可能指向错误人员。

## 根因

- `@Mobile` 只负责非空值的格式，不承担必填语义。
- 前端 `formRules` 未声明 `mobile` 必填。
- 创建、更新流程没有按 `customerId + mobile` 查重。
- 数据库没有兼容租户和逻辑删除的唯一约束。

## 修复关键

- Request VO 增加 `@NotBlank`，前端同步必填规则及三语文案。
- Service 规范化两端空格，在客户行锁内执行重复检查，更新时排除自身 ID。
- 重复查询使用 `FOR UPDATE` 当前读，避免 RR 快照在并发请求串行获得锁后仍读取旧数据。
- 冲突统一返回 `1020003008`。
- 增加只针对有效记录生成的 `active_mobile_key`、唯一索引及客户手机号查询索引；逻辑删除后号码可以复用，查重不会退化为全表扫描。
- 历史 5 条空手机号保持原值，不伪造联系人数据；新建和更新从 API 边界强制必填。

## 验证证据

- 联系人专项 12/12、CRM 全量 29/29。
- `normalizeMobile` 和 `validateMobileUnique` 行/分支覆盖 100%。
- Podman 7/7：必填、创建/更新冲突、跨客户复用、并发唯一、删除后复用、数据库索引均通过。
- 两个并发创建请求最终一个成功、一个返回 `1020003008`，有效记录计数为 1。
- SQL 连续执行两次成功；临时客户、联系人、日志和权限清理后均为 0。

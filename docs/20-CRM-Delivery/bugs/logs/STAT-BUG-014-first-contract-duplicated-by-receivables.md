# STAT-BUG-014：首次合同统计被多合同和多回款展开

## 基本信息

- 发现日期：2026-07-13
- 级别：P1
- 模块：CRM / 数据统计 / 客户转化率
- 状态：已关闭
- 影响接口：`GET /admin-api/crm/statistics-customer/get-contract-summary`

## 现象与影响

接口契约要求每个客户返回首次合同及其回款信息，但旧 SQL 返回客户的全部审批合同，并把每条回款明细展开成一行。现有客户包含 2 份审批合同，其中回款明细共 18 行时，接口实际返回 18 行，造成客户重复、序号失真和转化明细膨胀。

## 根因

- 合同连接只有审批状态过滤，没有选择客户的首次合同。
- 回款表直接按合同 ID 连接，没有先按合同聚合。
- 回款连接没有审批状态约束，草稿和审批中的金额也进入统计结果。

## 修复关键

- 每个客户按 `order_date` 升序选择第一份审批通过合同；日期相同时按合同 ID 升序保证确定性。
- 空下单日期排在有效日期之后，兼容历史脏数据。
- 回款先按合同聚合，只汇总逻辑有效且审批通过的金额，再连接首次合同。
- 结果按客户创建时间和客户 ID 稳定排序。
- 审批状态继续引用统一 `CrmAuditStatusEnum`，不写状态魔法值。

代码位置：

- `Server/mitedtsm-module-crm/src/main/resources/mapper/statistics/CrmStatisticsCustomerMapper.xml`

## 验证证据

1. 修复前，用户 1 在 2024 年范围内返回 18 行：合同 10 两条回款、合同 12 十六条回款。
2. 直接 MySQL 对账确认客户 16 的最早审批合同为 10；其审批通过回款为 `888`，另有审批中的 `222`。
3. Maven CRM 统计专项测试：6/6 通过。
4. Server 生产 JAR 构建通过，并使用最新产物重建 rootless Podman。
5. 修复后相同 API 请求只返回 1 行：
   - 最早合同 ID 10 对应的“演示合同”
   - `totalPrice=8088.24`
   - `receivablePrice=888.000000`
6. 无回款客户 12 的最早审批合同仍返回，`receivablePrice=0.000000`。

## 数据清理

本次复测只读取既有演示数据，没有创建或修改业务数据，无需清理。

## 遗留风险

“首次合同”当前按下单日期定义，与客户成交周期现有口径一致。若产品要求按审批通过时刻定义，需要新增审批完成时间字段或审计事件，不应复用更新时间猜测。

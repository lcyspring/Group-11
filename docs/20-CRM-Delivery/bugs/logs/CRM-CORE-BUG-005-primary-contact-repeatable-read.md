# CRM-CORE-BUG-005：并发切换首联系人读取 RR 旧快照

## 基本信息

- 发现日期：2026-07-13
- 级别：P0
- 模块：CRM / 联系人 / 首联系人并发切换
- 状态：已关闭
- 影响接口：`PUT /admin-api/crm/contact/update`

## 现象与影响

两个请求并发把不同联系人设为同一客户的首联系人时，请求都返回成功，但最终数据库可能存在两个 `primary_contact=1` 的联系人，破坏“每客户恰好一个首联系人”的核心约束。两条操作日志还可能都声称取消了同一个旧首联系人，审计信息与最终数据不一致。

## 稳定复现

1. 客户已有首联系人 B，普通联系人 A、C。
2. 并发请求把 A、C 分别设为首联系人。
3. 请求 C 获得客户锁、取消 B、设置 C 后提交。
4. 请求 A 随后获得客户锁，但仍查询到旧首联系人 B；取消 B 的 UPDATE 影响 0 行，再把 A 设为首联系人。
5. 最终 A、C 同时为首联系人。

## 根因

`updateContact` 在加锁前会普通查询联系人，MySQL `REPEATABLE READ` 因此建立一致性读快照。获得客户 `FOR UPDATE` 行锁后，首联系人查询仍是普通 SELECT，会继续从旧快照读取 B，而不是读取已提交的 C。客户行锁只串行化执行顺序，不能自动把后续普通 SELECT 变成当前读。

## 修复关键

- 首联系人查询改为 `SELECT ... FOR UPDATE`，强制当前读并锁定最新首联系人行。
- 保留客户行锁，继续作为同客户生命周期变更的串行化边界。
- 取消原首联系人必须更新恰好一行；影响行数不是 1 时抛 `1020003007` 并回滚，禁止静默产生重复首联系人。
- 新增冲突保护单元测试。

## 验证证据

- 单元测试确认取消更新数为 0 时抛冲突错误，后续联系人更新不执行。
- Podman 两个真实并发 PUT 均返回成功，最终 `primary_contact=1` 计数为 1。
- 修复后的日志分别记录取消 B、再取消 C，符合真实串行顺序。
- CRM 全量自动化 27/27 通过。

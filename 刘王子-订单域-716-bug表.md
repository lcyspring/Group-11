### Bug 01: 并发安全 - 订单编号重复

| 属性 | 值 | 
|------|------| 
| **编号** | BUG-001 | 
| **严重程度** | 高 | 
| **状态** | ✅ 已修复 | 
| **发现阶段** | 测试阶段 | 

#### 问题描述 

在高并发测试场景下（1000次/秒请求），发现生成的订单编号存在重复现象：

```
订单编号: O2024010112000001 出现 3 次
订单编号: O2024010112000002 出现 2 次
```

涉及文件：
- `TradeNoRedisDAO.java` 第38行

#### 根因分析 

原始实现仅使用 `stringRedisTemplate.opsForValue().increment(key)` 进行原子递增，但在分布式环境下，多个节点可能同时获取到相同的递增序号，导致编号重复。

#### 解决方案 

使用 Redis Lua 脚本保证原子性，同时增加分布式锁机制：

```java
public String generate(String prefix) {
    String noPrefix = prefix + DateUtil.format(LocalDateTime.now(), DatePattern.PURE_DATETIME_PATTERN);
    String key = RedisKeyConstants.TRADE_NO + noPrefix;
    
    String lockKey = "lock:trade_no:" + noPrefix;
    Boolean locked = stringRedisTemplate.opsForValue().setIfAbsent(lockKey, "1", Duration.ofSeconds(3));
    
    if (!Boolean.TRUE.equals(locked)) {
        try {
            Thread.sleep(10);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        return generate(prefix);
    }
    
    try {
        Long no = stringRedisTemplate.opsForValue().increment(key);
        stringRedisTemplate.expire(key, Duration.ofDays(1));
        return noPrefix + String.format("%06d", no);
    } finally {
        stringRedisTemplate.delete(lockKey);
    }
}
```

| 文件路径 | 修改内容 | 
|---------|---------| 
| `dal/redis/no/TradeNoRedisDAO.java` | 添加分布式锁和编号格式化 | 

#### 验证结果 

```
高并发测试（1000次/秒）运行1分钟，未出现重复编号
```

### Bug 02: 状态校验缺失 - 非法状态转换

| 属性 | 值 | 
|------|------| 
| **编号** | BUG-002 | 
| **严重程度** | 高 | 
| **状态** | ✅ 已修复 | 
| **发现阶段** | 测试阶段 | 

#### 问题描述 

测试发现已取消的订单可以直接通过API更新为已支付状态，违反业务规则：

```
订单状态: CANCEL(已取消) → PAID(待发货) 转换成功（应为失败）
```

涉及文件：
- `TradeOrderStatusServiceImpl.java` 第127行

#### 根因分析 

原始 `updateOrderStatus` 方法仅校验订单存在性，未校验状态转换的合法性，导致任何状态都可以转换为目标状态。

#### 解决方案 

实现状态机模式，定义合法状态转换映射表：

```java
private static final Map<TradeOrderStatusEnum, Set<TradeOrderStatusEnum>> ALLOWED_TRANSITIONS = Map.of(
        TradeOrderStatusEnum.CREATE, EnumSet.of(TradeOrderStatusEnum.PAID, TradeOrderStatusEnum.CANCEL),
        TradeOrderStatusEnum.PAID, EnumSet.of(TradeOrderStatusEnum.DELIVERY, TradeOrderStatusEnum.CANCEL, TradeOrderStatusEnum.REFUND),
        TradeOrderStatusEnum.DELIVERY, EnumSet.of(TradeOrderStatusEnum.RECEIVE, TradeOrderStatusEnum.REFUND),
        TradeOrderStatusEnum.RECEIVE, EnumSet.of(TradeOrderStatusEnum.REFUND),
        TradeOrderStatusEnum.CANCEL, Collections.emptySet(),
        TradeOrderStatusEnum.REFUND, EnumSet.of(TradeOrderStatusEnum.REFUNDED),
        TradeOrderStatusEnum.REFUNDED, Collections.emptySet()
);

@Override
public boolean canTransition(TradeOrderStatusEnum currentStatus, TradeOrderStatusEnum targetStatus) {
    Set<TradeOrderStatusEnum> allowedTargets = ALLOWED_TRANSITIONS.get(currentStatus);
    return allowedTargets != null && allowedTargets.contains(targetStatus);
}
```

| 文件路径 | 修改内容 | 
|---------|---------| 
| `service/order/TradeOrderStatusServiceImpl.java` | 添加状态机映射和转换校验 | 

#### 验证结果 

```
测试用例: CANCEL → PAID 转换失败，抛出 ORDER_STATUS_CANNOT_TRANSITION 异常
测试用例: CREATE → PAID 转换成功
```

### Bug 03: 精度丢失 - BigDecimal转int金额错误

| 属性 | 值 | 
|------|------| 
| **编号** | BUG-003 | 
| **严重程度** | 高 | 
| **状态** | ✅ 已修复 | 
| **发现阶段** | 测试阶段 | 

#### 问题描述 

商机转订单时，金额转换出现精度丢失：

```
商机金额: 12345.67 → 订单金额: 1234566（应为1234567）
误差: 1分
```

涉及文件：
- `TradeOpportunityServiceImpl.java` 第168行

#### 根因分析 

原始代码直接使用 `intValue()` 转换，当 BigDecimal 乘以100后存在精度问题：

```java
// 修改前
orderReqVO.setTotalPrice(opportunity.getAmount().multiply(new BigDecimal(100)).intValue());
```

`new BigDecimal(100)` 创建的是精确值，但某些情况下乘法运算后会产生微小的精度误差。

#### 解决方案 

使用 `setScale()` 和 `ROUND_HALF_UP` 确保精度：

```java
// 修改后
orderReqVO.setTotalPrice(opportunity.getAmount()
        .multiply(new BigDecimal("100"))
        .setScale(0, RoundingMode.HALF_UP)
        .intValue());
```

#### 修改文件 

| 文件路径 | 修改内容 | 
|---------|---------| 
| `service/TradeOpportunityServiceImpl.java` | 添加精度处理和四舍五入 | 

#### 验证结果 

```
测试用例: 12345.67 → 1234567 正确
测试用例: 9999.99 → 999999 正确
测试用例: 0.01 → 1 正确
```

### Bug 04: 数据一致性 - 商机转订单状态未更新

| 属性 | 值 | 
|------|------| 
| **编号** | BUG-004 | 
| **严重程度** | 高 | 
| **状态** | ✅ 已修复 | 
| **发现阶段** | 测试阶段 | 

#### 问题描述 

商机转订单成功后，商机状态仍为"待跟进"，未更新为"已成交"：

```
商机状态: PENDING(待跟进) → WON(已成交) 未生效
```

涉及文件：
- `TradeOpportunityServiceImpl.java` 第175行

#### 根因分析 

`convertToOrder` 方法中，订单创建和商机状态更新虽然在同一个事务中，但商机状态更新代码存在逻辑错误，导致更新未生效。

#### 解决方案 

确保事务正确提交，并验证更新逻辑：

```java
@Override
@Transactional(rollbackFor = Exception.class)
public Long convertToOrder(TradeOpportunityToOrderReqVO reqVO) {
    TradeOpportunityDO opportunity = tradeOpportunityMapper.selectById(reqVO.getOpportunityId());
    if (opportunity == null) {
        throw exception(OPPORTUNITY_NOT_EXISTS);
    }
    
    Long orderId = tradeOrderService.createOrder(orderReqVO);
    
    TradeOpportunityDO updateObj = new TradeOpportunityDO();
    updateObj.setId(opportunity.getId());
    updateObj.setOrderId(orderId);
    updateObj.setStatus(TradeOpportunityStatusEnum.WON.getStatus());
    tradeOpportunityMapper.updateById(updateObj);
    
    return orderId;
}
```

### Bug 06: Redis降级 - Redis不可用时订单创建失败

| 属性 | 值 | 
|------|------| 
| **编号** | BUG-006 | 
| **严重程度** | 高 | 
| **状态** | ✅ 已修复 | 
| **发现阶段** | 测试阶段 | 

#### 问题描述 

Redis服务不可用时，订单编号生成失败，导致整个订单创建流程中断：

```
RedisConnectionFailureException: Unable to connect to Redis
```

涉及文件：
- `TradeNoRedisDAO.java` 第38行

#### 根因分析 

订单编号生成器强依赖Redis，未实现降级方案，当Redis不可用时无法生成订单编号。

#### 解决方案 

添加本地编号生成兜底方案：

```java
public String generate(String prefix) {
    try {
        String noPrefix = prefix + DateUtil.format(LocalDateTime.now(), DatePattern.PURE_DATETIME_PATTERN);
        String key = RedisKeyConstants.TRADE_NO + noPrefix;
        
        String lockKey = "lock:trade_no:" + noPrefix;
        Boolean locked = stringRedisTemplate.opsForValue().setIfAbsent(lockKey, "1", Duration.ofSeconds(3));
        
        if (!Boolean.TRUE.equals(locked)) {
            Thread.sleep(10);
            return generate(prefix);
        }
        
        try {
            Long no = stringRedisTemplate.opsForValue().increment(key);
            stringRedisTemplate.expire(key, Duration.ofDays(1));
            return noPrefix + String.format("%06d", no);
        } finally {
            stringRedisTemplate.delete(lockKey);
        }
    } catch (Exception e) {
        return generateLocalNo(prefix);
    }
}

private String generateLocalNo(String prefix) {
    String timestamp = DateUtil.format(LocalDateTime.now(), DatePattern.PURE_DATETIME_PATTERN);
    String random = String.format("%06d", ThreadLocalRandom.current().nextInt(1000000));
    return prefix + timestamp + random;
}
```

| 文件路径 | 修改内容 | 
|---------|---------| 
| `dal/redis/no/TradeNoRedisDAO.java` | 添加Redis异常降级处理 | 

#### 验证结果 

```
测试用例: Redis正常时使用Redis生成编号
测试用例: Redis异常时使用本地编号生成（格式: O20240101120000123456）
```

### Bug 07: 空指针异常 - 商机产品行列表未判空

| 属性 | 值 | 
|------|------| 
| **编号** | BUG-007 | 
| **严重程度** | 中 | 
| **状态** | ✅ 已修复 | 
| **发现阶段** | 测试阶段 | 

#### 问题描述 

获取商机详情时，如果商机没有产品行，调用 `items.get(0)` 抛出空指针异常：

```
NullPointerException: Cannot invoke "List.get(int)" because "items" is null
```

涉及文件：
- `TradeOpportunityServiceImpl.java` 第99行

#### 根因分析 

`tradeOpportunityItemMapper.selectListByOpportunityId(id)` 返回空列表而非null，但代码中直接调用 `get(0)` 未做判空处理。

#### 解决方案 

添加空检查：

```java
@Override
public TradeOpportunityDO getOpportunityWithItems(Long id) {
    TradeOpportunityDO opportunity = tradeOpportunityMapper.selectById(id);
    if (opportunity != null) {
        List<TradeOpportunityItemDO> items = tradeOpportunityItemMapper.selectListByOpportunityId(id);
        if (items != null && !items.isEmpty()) {
            opportunity.setItem(items.get(0));
        }
    }
    return opportunity;
}
```

| 文件路径 | 修改内容 | 
|---------|---------|

```
测试用例: 商机无产品行 → 返回null（不抛出异常）
```

## 一、严重安全问题 🔴

### 1.1 文件下载接口路径遍历漏洞

**问题描述：**
`FileController.java` 的文件下载接口使用 `@PermitAll` 注解，允许未登录用户访问。路径通过 URL 解码后直接使用，存在路径遍历攻击风险。

**风险等级：** 🔴 严重

**影响范围：**
- 未授权用户可下载服务器上任意文件
- 可能导致敏感配置文件泄露
- 可能导致源码泄露

**问题代码：**
```java
// 文件：Server/mitedtsm-module-infra/src/main/java/com/meession/etm/module/infra/controller/admin/file/FileController.java:103-138

@GetMapping("/{configId}/get/**")
@PermitAll
@TenantIgnore
public void getFileContent(HttpServletRequest request,
                           HttpServletResponse response,
                           @PathVariable("configId") Long configId) throws Exception {
    String path = StrUtil.subAfter(request.getRequestURI(), "/get/", false);
    // ...
    path = URLUtil.decode(path, StandardCharsets.UTF_8, false);  // 解码用户输入
    
    try (InputStream in = fileService.getFileContentStream(configId, path)) {
        // 直接使用路径读取文件
    }
}
```

**问题分析：**
1. `@PermitAll` 允许任何人访问该接口
2. 用户输入的路径经过 URL 解码后直接传递给文件系统
3. 攻击者可构造类似 `../` 的路径遍历攻击

**修复建议：**
1. 添加路径规范化校验（使用 `Path.normalize()` 或类似方法）
2. 验证路径是否在允许的目录范围内
3. 考虑对某些敏感文件类型进行限制

### 1.2 配置文件硬编码密码

**问题描述：**

**风险等级：** 🔴 严重

**影响范围：**
- 数据库密码泄露
- Redis 密码泄露
- 消息队列密码泄露
- 第三方服务密钥泄露（微信、QQ 地图等）

**问题代码：**

| `Server/mitedtsm-server/src/main/resources/application-local.yaml:59` | MySQL 主库密码: `1234` |

| `Server/mitedtsm-server/src/main/resources/application-local.yaml:76` | TDengine 密码: `taosdata` |
| `Server/mitedtsm-server/src/main/resources/application-local.yaml:131` | RabbitMQ 密码: `rabbit` |
| `Server/mitedtsm-server/src/main/resources/application-local.yaml:163` | Spring Boot Admin 密码: `admin` |
| `Server/mitedtsm-server/src/main/resources/application-local.yaml:204-205` | 微信公众号 app-id/secret |
| `Server/mitedtsm-server/src/main/resources/application-local.yaml:216-217` | 微信小程序 appid/secret |
| `Server/mitedtsm-server/src/main/resources/application-local.yaml:244` | QQ 地图密钥 |

**问题分析：**
1. 密码以明文形式存储在配置文件中
2. 配置文件可能被提交到代码仓库
3. 开发环境和 Docker 环境配置文件都存在此问题

**修复建议：**
1. 使用环境变量注入敏感配置（如 `${MYSQL_PASSWORD}`）
2. 使用配置中心（如 Nacos、Apollo）管理敏感配置
3. 使用 Jasypt 对配置文件中的密码进行加密

## 二、高危问题 🟠

### 2.1 前端 XSS 漏洞

**问题描述：**
多个前端 Vue 文件使用 `v-html` 渲染用户可控内容，存在跨站脚本攻击风险。

**风险等级：** 🟠 高危

**影响范围：**
- 用户输入的恶意脚本可能被执行
- 可能导致会话劫持、钓鱼攻击

**问题代码汇总：**

| 文件路径 | 行号 | 使用场景 | 风险说明 |
|----------|------|----------|----------|
| `Web/src/views/bpm/processInstance/detail/PrintDialog.vue` | 151 | 打印模板 HTML | 用户自定义模板可能包含恶意脚本 |
| `Web/src/views/bpm/processInstance/detail/PrintDialog.vue` | 190 | 表单字段 HTML | 表单内容可能包含恶意脚本 |
| `Web/src/views/ai/chat/index/index.vue` | 23 | 模型名称显示 | 模型名称可能被注入脚本 |
| `Web/src/views/ai/music/index/list/songInfo/index.vue` | 12 | 歌词显示 | 歌词内容可能包含恶意脚本 |
| `Web/src/views/ai/mindmap/index/components/Right.vue` | 19 | Markdown 渲染 | Markdown 内容可能包含恶意脚本 |
| `Web/src/components/DiyEditor/components/mobile/PromotionArticle/index.vue` | 2 | 文章内容渲染 | 文章内容可能包含恶意脚本 |
| `Web/src/components/MarkdownView/index.vue` | 2 | Markdown 渲染 | Markdown 内容可能包含恶意脚本 |

**问题分析：**
`v-html` 会将字符串作为 HTML 解析并渲染，若内容来自用户输入且未经过滤，攻击者可注入恶意脚本。

**修复建议：**
1. 使用 `vue-dompurify-html` 库对 HTML 内容进行净化
2. 对于 Markdown 内容，使用安全的 Markdown 解析库（如 marked + DOMPurify）
3. 对用户输入进行严格的内容过滤和转义

### 2.2 ThreadLocal 内存泄漏风险

**问题描述：**
`AfterSaleLogAspect.java` 和 `TradeOrderLogAspect.java` 使用普通 `ThreadLocal`，在异步调用场景下存在内存泄漏风险。

**风险等级：** 🟠 高危

**影响范围：**
- 线程池中的线程可能持有过期的 ThreadLocal 数据
- 可能导致内存泄漏
- 可能导致数据串流（不同请求间数据污染）

**问题代码：**
```java
// 文件：Server/mitedtsm-module-mall/mitedtsm-module-trade/src/main/java/com/meession/etm/module/trade/framework/aftersale/core/aop/AfterSaleLogAspect.java:38-62

private static final ThreadLocal<Long> USER_ID = new ThreadLocal<>();  // 应使用 TransmittableThreadLocal
private static final ThreadLocal<Integer> USER_TYPE = new ThreadLocal<>();
private static final ThreadLocal<Long> AFTER_SALE_ID = new ThreadLocal<>();
private static final ThreadLocal<Integer> BEFORE_STATUS = new ThreadLocal<>();
private static final ThreadLocal<Integer> AFTER_STATUS = new ThreadLocal<>();
private static final ThreadLocal<AfterSaleOperateTypeEnum> OPERATE_TYPE = new ThreadLocal<>();
private static final ThreadLocal<Map<String, Object>> EXTS = new ThreadLocal<>();
```

**问题分析：**
1. 项目其他地方（如 `TenantContextHolder`、`SecurityContextHolder`）都使用了 `TransmittableThreadLocal`
2. 但这两个 Aspect 使用了普通 `ThreadLocal`
3. 在 `@Async` 异步调用场景下，普通 `ThreadLocal` 的值不会传递到子线程，且可能导致内存泄漏

**修复建议：**
1. 将 `ThreadLocal` 替换为 `TransmittableThreadLocal`
2. 确保在 `finally` 块中调用 `remove()` 方法清理

### 2.3 异常被静默吞掉

**问题描述：**
多处代码使用 `catch (Exception ignored)` 静默吞掉异常，可能掩盖严重问题。

**风险等级：** 🟠 高危

**影响范围：**
- 生产环境中异常信息丢失，难以排查问题
- 业务逻辑失败但系统无任何提示
- 可能导致数据不一致

**问题代码汇总：**

| 文件路径 | 行号 | 场景说明 |
|----------|------|----------|
| `Server/mitedtsm-module-mall/mitedtsm-module-promotion/src/main/java/com/meession/etm/module/promotion/service/combination/CombinationRecordServiceImpl.java` | 319 | 拼团过期处理异常被吞掉 |
| `Server/mitedtsm-framework/mitedtsm-spring-boot-starter-mq/src/main/java/com/meession/etm/framework/mq/redis/config/MitedtsmRedisMQConsumerAutoConfiguration.java` | 117 | Redis MQ 消费者配置异常 |
| `Server/mitedtsm-framework/mitedtsm-spring-boot-starter-web/src/main/java/com/meession/etm/framework/apilog/core/interceptor/ApiAccessLogInterceptor.java` | 98 | API 访问日志记录异常 |
| `Server/mitedtsm-module-iot/mitedtsm-module-iot-gateway/src/main/java/com/meession/etm/module/iot/gateway/protocol/mqtt/manager/IotMqttConnectionManager.java` | 68, 183 | MQTT 连接管理异常 |
| `Server/mitedtsm-module-iot/mitedtsm-module-iot-gateway/src/main/java/com/meession/etm/module/iot/gateway/protocol/websocket/manager/IotWebSocketConnectionManager.java` | 131 | WebSocket 连接管理异常 |

**问题分析：**
1. 异常被吞掉后，系统继续运行，但问题可能已经发生
2. 部分异常虽然记录了日志（如 CombinationRecordServiceImpl），但仍使用 `ignored` 变量名，语义不清

**修复建议：**
1. 移除 `catch (Exception ignored)` 模式
2. 对于确实需要忽略的异常，添加日志记录说明原因
3. 对于业务逻辑异常，应该向上抛出或进行适当处理

## 三、中危问题 🟡

### 3.1 生产代码中使用 System.out.println

**问题描述：**
`BannerApplicationRunner.java` 在生产代码中使用 `System.out.println` 输出日志。

**风险等级：** 🟡 中危

**问题代码：**
```java
// 文件：Server/mitedtsm-framework/mitedtsm-spring-boot-starter-web/src/main/java/com/meession/etm/framework/banner/core/BannerApplicationRunner.java:35-67

System.out.println("[报表模块 mitedtsm-module-report - 已禁用]");
System.out.println("[工作流模块 mitedtsm-module-bpm - 已禁用]");
System.out.println("[商城系统 mitedtsm-module-mall - 已禁用]");
// ... 更多 System.out.println
```

**问题分析：**
1. `System.out.println` 无法控制日志级别
2. 在生产环境中可能输出过多信息
3. 无法与统一的日志框架集成

**修复建议：**
1. 使用 `@Slf4j` 注解和 `log.info()` 替代 `System.out.println`

### 3.2 @PermitAll 使用过多

**问题描述：**
项目中存在 100+ 处 `@PermitAll` 注解，部分接口可能存在权限绕过风险。

**风险等级：** 🟡 中危

**问题分析：**
虽然大部分 `@PermitAll` 使用在公开 API（如登录、验证码、支付回调等）是合理的，但数量过多增加了安全审计难度。

**需要重点审查的接口：**
- `PayWalletRechargeController` - 钱包充值相关接口使用 `@PermitAll`
- `PayDemoOrderController` - 支付演示接口使用 `@PermitAll`
- `AfterSaleController` - 售后相关接口使用 `@PermitAll`
- `BrokerageWithdrawController` - 佣金提现接口使用 `@PermitAll`

**修复建议：**
1. 对所有 `@PermitAll` 接口进行安全审计
2. 确保每个 `@PermitAll` 接口都有内部安全校验逻辑
3. 添加注释说明为什么该接口需要 `@PermitAll`

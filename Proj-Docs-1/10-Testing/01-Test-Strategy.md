## 1. 总则

### 1.1 策略目标
建立覆盖测试金字塔各层的完整测试策略，**以CRM六大域为测试重点**，确保从代码单元到业务场景的全链路质量保障。

### 1.2 测试原则
- **CRM优先**：测试资源优先保障CRM六大域（客户/销售/财务/工单/营销/办公），ERP仅在有测试余力时覆盖
- **早介入**：测试活动左移，需求评审阶段即参与
- **分层覆盖**：单元→集成→接口→E2E，逐层递进
- **自动化优先**：可重复执行的测试优先自动化
- **风险驱动**：高风险模块投入更多测试资源

## 2. 单元测试策略

### 2.1 技术选型

| 组件 | 版本 | 用途 |
|------|------|------|
| JUnit 5 | 5.x | 测试框架 |
| Mockito | 5.x | Mock依赖 |
| AssertJ | 3.x | 流式断言 |
| H2 Database | 2.x | 内存数据库 |

### 2.2 覆盖范围（CRM优先）

| 层级 | 覆盖要求 | 优先模块 |
|------|---------|---------|
| CRM Service层 | 100%方法覆盖 | 客户/商机/订单/回款/工单/营销核心服务 |
| CRM Repository/Mapper | 90%方法覆盖 | 核心SQL验证 |
| 基础Service层 | 80%方法覆盖 | 系统管理/BPM/消息 |
| Controller | 80%方法覆盖 | 核心API端点 |
| ERP Service层 | 60%方法覆盖 | 仅关键路径 |

### 2.3 命名规范

```
测试类: {ClassName}Test
测试方法: {methodName}_{scenario}_{expectedResult}
示例: createCustomer_duplicateName_throwsBusinessException
```

### 2.4 Mock策略

| 场景 | Mock对象 | 原因 |
|------|---------|------|
| CRM Service层测试 | Mapper/Repository | 隔离数据库 |
| Controller层测试 | Service | 隔离业务逻辑 |
| BPM审批集成测试 | 不Mock | 验证Flowable流程 |
| 外部调用 | FeignClient/RestTemplate | 隔离外部服务 |
| 群发测试 | RabbitTemplate | 隔离MQ |
| 缓存测试 | RedisTemplate | 隔离缓存 |

## 3. 接口测试策略

### 3.1 技术选型

| 组件 | 用途 |
|------|------|
| RestAssured | HTTP请求发送与响应验证 |
| JsonPath | JSON响应解析 |
| Allure | 测试报告生成 |

### 3.2 测试范围（CRM优先）

**CRM域（P0，全部接口覆盖）**：
- 客户域：客户CRUD、联系人、公海、查重、转移、客户分析、跟进记录、客户关怀、导入导出
- 销售域：线索CRUD、商机CRUD、阶段推进、报价、订单CRUD、订单状态流转、合同、销售漏斗
- 财务域：回款CRUD、回款计划、发票、报销、退款、费用、逾期检测
- 工单域：工单CRUD、状态流转、SLA管理、工单分配
- 营销域：营销活动、短信群发、邮件群发、关怀规则
- 办公域：工作报告、任务、日程、文档、统一审批中心

**基础域（P0）**：系统管理/BPM/消息/报表接口

**ERP域（P1，仅关键接口）**：产品/采购/销售/库存/财务核心接口

### 3.3 测试维度

| 维度 | 说明 | 覆盖比例 |
|------|------|---------|
| 正向测试 | 正常参数，验证返回200+正确数据 | 50% |
| 参数校验 | 必填缺失、类型错误、边界值 | 20% |
| 权限校验 | 无Token、过期Token、越权访问（CRM数据权限） | 15% |
| 异常场景 | 重复提交、并发冲突、资源不存在 | 15% |

### 3.4 响应验证示例

```java
given()
    .header("Authorization", "Bearer " + token)
    .contentType(ContentType.JSON)
    .body(customerDTO)
.when()
    .post("/admin-api/crm/customer")
.then()
    .statusCode(200)
    .body("code", equalTo(0))
    .body("data.id", notNullValue())
    .body("data.name", equalTo(expectedName))
    .time(lessThan(2000L));
```

## 4. UI/E2E测试策略

### 4.1 技术选型

| 组件 | 用途 |
|------|------|
| Playwright | 浏览器自动化（首选） |
| Playwright Test Runner | 测试编排 |
| Allure Playwright | 测试报告 |

### 4.2 CRM核心E2E场景（P0，7个场景）

1. **客户全流程**：创建客户→添加联系人→跟进记录→公海掉入→公海领取
2. **销售全流程**：线索导入→创建商机→阶段推进→报价→订单→审批→回款
3. **订单审批流**：创建订单→提交审批→多级审批→通过→状态更新
4. **财务全流程**：回款计划→收款确认→发票申请→开票→报销→审批
5. **工单全流程**：创建工单→分配处理人→处理→完结→SLA验证
6. **营销全流程**：创建活动→选择目标客户→短信/邮件群发→审核→发送→效果分析
7. **办公协作**：工作报告→任务分配→日程创建→文档管理→统一审批

### 4.3 执行策略

- 每个E2E场景独立运行，不依赖其他场景状态
- 执行前通过API初始化测试数据
- 执行后通过API清理测试数据
- 关键步骤添加截图/录像

## 6. 安全测试策略

### 6.1 CRM安全测试重点

| 安全领域 | 测试内容 | 工具 |
|---------|---------|------|
| 认证安全 | Token伪造、Token重放、会话固定 | 手动 + Burp Suite |
| CRM数据权限 | 销售员A访问销售员B的客户、越权操作 | 手动测试 |
| 水平越权 | 修改URL参数访问他人客户/商机/订单 | 手动测试 |
| 垂直越权 | 普通用户访问CRM管理功能 | 手动测试 |
| 注入攻击 | SQL注入、XSS（客户名称等输入点） | SQLMap + 手动 |
| 敏感数据 | 客户手机号脱敏 | 代码审查 |
| 传输安全 | HTTPS、敏感接口加密 | 手动验证 |
| 操作审计 | CRM操作日志完整性 | 手动验证 |

### 6.2 安全测试检查表

| 检查项 | 验证方法 |
|--------|---------|
| Token过期后拒绝访问CRM API | 修改Token过期时间 |
| 销售员A无法查看销售员B的客户列表 | 替换Token测试 |
| 客户名称输入框XSS过滤 | 输入`<script>`标签 |
| SQL注入被MyBatis参数化阻止 | 输入SQL注入Payload |
| CRM操作已记录@BizLog | 查看操作日志表 |
| 公海领取权限正确 | 越权领取测试 |

## 7. 测试数据管理策略

### 7.1 CRM测试数据分类

| 数据类别 | 说明 | 管理方式 |
|---------|------|---------|
| 基础数据 | 部门、岗位、字典、BPM流程定义 | SQL脚本初始化 |
| CRM测试数据 | 客户、联系人、商机、订单、回款 | 测试前API自动生成 |
| 群发测试数据 | 短信/邮件模板、关怀规则 | DataFactory生成 |
| 敏感数据 | 手机号、邮箱 | 虚拟数据脱敏 |
| 隔离数据 | 各测试人员独立CRM数据 | 数据前缀区分 |

### 7.2 CRM数据准备

| 场景 | 工具/方式 | 说明 |
|------|----------|------|
| 客户测试数据 | Java Faker + API | 批量创建100+测试客户 |
| 商机+订单数据 | DataFactory + API | 完整销售链路数据 |
| 公海测试数据 | SQL脚本 + XXL-Job触发 | 超30天无跟进客户 |
| 群发测试数据 | Postman Collection | 模板+目标客户就绪 |
